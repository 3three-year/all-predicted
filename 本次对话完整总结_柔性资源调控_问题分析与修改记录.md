## 2. 项目总体内容与调度对象

当前项目是一个包含多类柔性资源的农村台区综合调度环境，主要调度对象包括：

1. 储能
2. 空调柔性负荷
3. EV（电动汽车）负荷
4. 可平移负荷

强化学习算法通过离散动作对上述资源进行调节，当前主要动作大致对应为：

- `action == 0`：储能放电相关
- `action == 1`：储能充电相关
- `action == 2`：空调调节
- `action == 3`：EV 调节
- `action == 4`：可平移负荷调节

主要比较算法包括：

- `Baseline`
- `DDQN`
- `T-DDQN`
- `Ablation`
- `PPO`

## 3. 当前成本指标与电价设计

### 3.1 成本指标定义

当前典型日成本的计算方式为：

```text
Cost = Σ [ max(0, load(t) - pv(t)) × price(t) × 0.25 ]
```

其中：

- `load(t)`：当前算法在时刻 `t` 下形成的总负荷
- `pv(t)`：当前时刻光伏出力
- `price(t)`：当前时刻电价
- `0.25`：15 分钟对应的小时长度

### 3.2 成本指标物理含义

这个指标表示：

**当天从电网购电所需支付的总运行成本。**

它只计算购电成本，不考虑：

- 余电上网收益
- 售电收入
- 负成本收益

所以它本质上是：

**购电成本指标**

而不是完整的系统经济净收益。

## 4. 电价设计的讨论与修改

### 4.1 原始电价设计

项目最初的电价并不是固定分时电价，而是一个状态反馈型动态电价，包含：

- `base_price`
- `seasonal_factor`
- `day_factor`
- `load_factor`
- `pv_factor`
- `soc_factor`

即：

```text
price = base_price
      × seasonal_factor
      × day_factor
      × load_factor
      × pv_factor
      × soc_factor
```

### 4.2 原始电价设计的问题

这种设计会导致：

- 算法的行为反过来影响价格环境
- 各算法面对的价格轨迹并不完全一致
- 不利于做“同价格环境下的公平成本对比”

尤其不合理的包括：

- `load_factor`
- `pv_factor`
- `soc_factor`
- `day_factor`

其中：

- `load_factor`：把负荷结果又反馈进价格
- `pv_factor`：把光伏消纳结果又反馈进价格
- `soc_factor`：把储能状态又反馈进价格

这些会让“算法优劣”和“算法自己改变了价格”混在一起。

### 4.3 最终修改结果

后来已将电价函数简化成固定分时电价：

- 峰时：`8-11`, `18-22`
- 谷时：`23-7`
- 平时：其余时段取峰谷均值

现在：

```text
price = base_price
```

这样所有算法面对同一外生价格环境，更适合比较：

**不同调度策略对净负荷与成本的真实影响。**

## 5. EV 模型：讨论、修改与当前状态

EV 是本轮修改中最系统、最完整的一块。

### 5.1 原始 EV 放电逻辑问题

原来 EV 放电主要依赖固定绝对阈值：

```text
pv_output < 1.0
current_load > 0.8
ev_load.soc > 0.4
```

问题包括：

1. 固定绝对阈值不适应不同日型
2. 没有直接体现“净负荷压力”
3. 工程规则意味太强，不利于论文表达

### 5.2 EV 放电逻辑修改

将 EV 放电改成基于净负荷相对超额程度的连续判断。

定义：

```text
L_net = current_load - pv_output
L_base = day_mean_net_load
L_peak = day_peak_net_load
```

触发条件：

```text
L_net > (1 + eta_L) * L_base
and SOC > 0.50
```

放电强度：

```text
net_load_ratio = clip((L_net - L_base) / (L_peak - L_base), 0, 1)
Delta_P_peak = - min(discharge_limit, k_discharge * |P_ev_min| * net_load_ratio)
```

当前讨论和实现中常用参数：

- `eta_L = 0.10`
- `k_discharge = 0.50`
- `SOC_dis_threshold = 0.50`

这一步的意义是：

**把 EV 放电从“低光伏 + 高负荷”的固定规则，改成了“净负荷越高于基准，放电越强”的连续削峰机制。**

### 5.3 EV 充电逻辑修改

原来 EV 充电分场景判断依赖固定绝对阈值：

- 高光伏：`pv_output >= 3.0`
- 中光伏：`1.0 <= pv_output < 3.0`
- 低光伏：`pv_output < 1.0`

后面改成按相对光伏比例判断：

```text
day_pv_peak = max(day_pv_profile)
pv_ratio = pv_output / max(day_pv_peak, 1e-6)
```

并划分为：

- 高光伏：`pv_ratio >= 0.50`
- 中光伏：`0.15 <= pv_ratio < 0.50`
- 低光伏：`pv_ratio < 0.15`

这样更适应不同天气、季节和光伏倍率。

### 5.4 EV 奖励层统一

原来 EV 奖励层还残留旧的固定阈值：

- `pv_output > 5.0`
- `pv_output < 1.0`
- `pv_output < 1.5`

后来统一改成相对光伏比率判断：

- 高光伏奖励：`reward_pv_ratio >= 0.50`
- 低光伏惩罚：`reward_pv_ratio < 0.15`
- 更低光伏惩罚：`reward_pv_ratio < 0.10`

这样 EV 决策逻辑和奖励逻辑口径统一。

### 5.5 EV 目标功率结构化表达

后续讨论中，将 EV 目标功率整理成半结构化形式：

```text
P_ev_target = P_task + Delta_P_pv + Delta_P_prog + Delta_P_peak
```

含义为：

- `P_task`：任务底线项
- `Delta_P_pv`：光伏吸纳修正项
- `Delta_P_prog`：进度补偿项
- `Delta_P_peak`：削峰放电修正项

这一步非常重要，因为它把原来散在 if-else 里的规则变成了可解释结构。

### 5.6 EV fallback 机制

为了让 DDQN / PPO 在测试阶段也能显示出 EV 调节曲线，增加了：

```python
enable_ev_fallback
```

默认：

```text
False
```

在测试阶段可显式打开，使 DDQN / PPO 即使没主动选 `action == 3`，也能有 EV 保底调节行为。

### 5.7 EV 当前结果与问题

从这次结果看，EV 是三类资源中最成熟的一块：

- 光伏吸纳明显
- 白天调节积极
- 任务完成效果好

但当前仍有几个问题：

1. 中午充电过强，可能抬高中午负荷
2. DDQN / PPO 的 EV 行为包含 fallback 成分
3. EV 放电削峰作用还不如充电吸纳作用突出

## 6. 储能模型：讨论、修改与当前状态

储能是本轮对话中第二大重点。

### 6.1 储能原始角色

储能在项目中承担多个作用：

1. 光伏富余吸纳
2. 谷电充电
3. 晚高峰放电
4. 削峰填谷
5. 成本优化

### 6.2 关于“储能为什么会充电”的讨论

我们明确了一个物理事实：

**储能不是发电机，不能凭空放电，必须先充电后放电。**

所以：

- 夜间/凌晨在谷价时段充电本身是合理的
- 中午光伏高时充电也合理

问题不在“为什么会充电”，而在：

**充进去以后后面有没有在该放的时候放出来。**

### 6.3 储能功率图：从 SOC 反推改成直接记录原始功率

最初的储能功率图是根据 SOC 反推的：

```text
P_bat(t) = [SOC(t) - SOC(t-1)] * E_bat / Δt
```

这能反映大致趋势，但问题很多：

- 容易放大 SOC 跳变
- 更像等效功率而不是原始功率
- 解释不够严谨

后来改成在环境 `step()` 中直接记录：

```python
self.battery_power_kw
```

并在测试阶段由 trainer 直接采集，生成：

- `07a_典型日1储能功率曲线_09-16.png`
- `07b_典型日2储能功率曲线_09-25.png`

现在这两张图是原始记录功率，不再是 SOC 差分反推图。

### 6.4 储能“重充轻放”问题

这轮对话中最早暴露的问题是：

- 储能会充电
- SOC 会升高
- 但后面很少放电

原因逐步被定位为：

1. 充电入口多
2. 放电入口少
3. 放电主要依赖 `action == 0`
4. 动作层不爱选 `0`
5. 部分放电逻辑门槛太严格

### 6.5 放电驱动增强：讨论与修改

后续给储能增加了放电驱动，包括：

#### 1. 放电触发条件

从旧逻辑逐步改成更偏相对净负荷的判断：

```text
net_load_now > 1.05 * day_mean_net_load
and SOC > 0.35
```

#### 2. 放电奖励

增加了显式放电奖励，鼓励在高净负荷/高价时段放电。

#### 3. 高 SOC 不放电惩罚

引入“高 SOC + 高净负荷 + 较高电价却不放”的机会成本惩罚。

### 6.6 储能放电 fallback

后来发现真正打到根上的问题是：

**储能保底放电逻辑之前被错误地写进了 `action == 2` 分支。**

这意味着：

- 只有动作刚好为 2 时，fallback 才可能触发
- 它并没有对所有 RL 算法普遍生效

后续修正为独立逻辑：

```text
if algo != baseline
and action != 0
and battery_soc > threshold
and battery_net_load_ratio > threshold
and price_ratio > threshold:
    fallback_discharge
```

并逐步把这些门槛放宽：

- `battery_soc` 从 `0.70`、`0.60` 逐步讨论并放宽到 `0.40`
- `price_ratio` 从 `0.60` 放宽到 `0.45`
- `battery_net_load_ratio` 从 `0.50` 放宽到 `0.35`

### 6.7 当前储能结果

从最新图看：

- 典型日 1：`DDQN / PPO` 仍然偏弱，`Baseline` 放电最明显
- 典型日 2：`DDQN / T-DDQN / PPO` 已能在晚间出现放电
- `Ablation` 的储能几乎一直闲置

说明储能已经从：

```text
几乎只充不放
```

发展到：

```text
部分日型可以放电，但仍不稳定
```

当前最大问题是：

1. 放电还不够稳定
2. 仍然带有 fallback 驱动痕迹
3. 典型日 1 仍然偏弱
4. Ablation 几乎不使用储能

## 7. 空调模型：当前实现、公式、动作调控与遗留问题

空调是当前三类资源中**最老、最工程化、最不规整**的一块。

### 7.1 空调模型本体

空调类是：

- `AirConditioningLoad`

继承自：

- `FlexibleLoad`

其核心参数包括：

- `max_power = 15.0`
- `min_power = -10.0`
- `response_time = 3`
- `ramp_rate = 2.0`
- `comfort_temp = 26.0`

### 7.2 功率更新公式

目标功率裁剪：

```text
P_ac,target = clip(P_target, P_min, P_max)
```

实际功率更新：

```text
power_diff = target_power - current_power
max_delta = ramp_rate * delta_time / response_time
delta = clip(power_diff, -max_delta, max_delta)
current_power = current_power + delta
```

### 7.3 温度更新公式

代码含义近似为：

```text
temp_change_rate = -0.1 * (current_power / max_power)
                   + 0.05 * (outdoor_temp - current_temp)

current_temp = current_temp + temp_change_rate * delta_time
```

并限制：

```text
16°C <= current_temp <= 30°C
```

### 7.4 `action == 2` 的调控逻辑

空调动作是：

```python
action == 2
```

当前这部分逻辑高度工程化，主要根据：

- `pv_output`
- `current_load`
- `hour / time_step`
- 算法类型（特别是 `T-DDQN`）

来决定目标功率。

#### T-DDQN 空调逻辑中的典型规则

高光伏时：

```text
if pv_output > 3.0:
    if current_load > 1.2:
        pv_factor = min(0.8, pv_output / 15.0)
        target_power = max_power * (0.5 + 0.6 * pv_factor)
    else:
        pv_factor = min(0.9, pv_output / 15.0)
        target_power = max_power * (0.6 + 0.7 * pv_factor)
```

低光伏时：

```text
if pv_output < 1.0:
    if current_load > 0.5:
        reduction_factor = min(0.7, (current_load - 0.5) / 1.5)
        max_reduction = min(load_demand * 0.4, max_power * 0.6)
        target_power = - max_reduction * reduction_factor
    else:
        target_power = - max_power * 0.3
```

中低光伏时：

```text
elif pv_output < 2.0:
    if current_load > 0.4:
        reduction_factor = min(0.8, (current_load - 0.4) / 1.0)
        max_reduction = min(load_demand * 0.5, max_power * 0.7)
        target_power = - max_reduction * reduction_factor
    else:
        target_power = - max_power * 0.4
```

还叠加了晚间更强削减逻辑，以及额外的负荷压缩分支。

#### 非 T-DDQN 的空调逻辑

高光伏时：

```text
if pv_output > 4.0:
    if current_load > 2.0:
        target_power = max_power * (0.2 + 0.4 * pv_factor)
    else:
        target_power = max_power * (0.3 + 0.5 * pv_factor)
```

负荷较高时削减：

```text
if current_load > 1.5:
    reduction_factor = min(0.6, (current_load - 1.5) / 2.5)
    max_reduction = min(load_demand * 0.3, max_power * 0.6)
    target_power = - max_reduction * reduction_factor
else:
    target_power = max_power * 0.15
```

### 7.5 空调中的平滑处理

空调分支最后通常还会进行平滑：

```text
target_power = alpha * target_power + (1 - alpha) * prev_ac_power
```

其中：

- `alpha = 0.2`

并限制单步变化：

```text
max_change = max_power * 0.2
```

### 7.6 当前空调中的旧阈值

空调逻辑里当前仍然残留大量绝对阈值：

#### 光伏阈值

- `pv_output > 3.0`
- `pv_output > 4.0`
- `pv_output < 1.0`
- `pv_output < 2.0`

#### 负荷阈值

- `current_load > 0.2`
- `0.3`
- `0.4`
- `0.5`
- `1.0`
- `1.2`
- `1.5`
- `2.0`

#### 峰时定义

空调内部还存在一套自己的峰时概念：

```text
0 <= hour <= 6 or 18 <= hour <= 24
```

而当前电价峰时是：

- `8-11`
- `18-22`

这两套并不一致。

### 7.7 当前空调动作层面的启发式

在 `DRL.py` 中，`T-DDQN` 与 `Ablation` 的动作选择层大量高概率返回：

```text
[2, 3, 4]
```

这意味着：

- 空调动作不是完全由策略自由学出
- 而是被启发式强烈推动

### 7.8 当前空调最主要的问题

1. 旧阈值残留最多
2. 绝对值逻辑多，和 EV / 储能脱节
3. 峰时定义与电价不一致
4. 动作启发式依赖强
5. 容易与 EV / 储能在中午形成重复吸纳
6. 是当前最值得继续结构化重构的模块

## 8. 结果图综合解释

- 晚高峰净负荷较高

说明储能问题已经从：

```text
完全不会放电
```

转变成：

```text
部分算法、部分日型会放，但稳定性不足
```

### 8.5 EV 图

说明 EV 已经是最积极的柔性资源：

- 白天长时间充电
- 中午显著吸纳光伏

但也意味着它对中午负荷抬升作用较强。

## 10. 本轮对话中涉及的重要代码层面问题

在修改过程中，反复定位和修复过一些工程问题，包括：

1. `enable_ev_fallback` 属性缺失
2. `mode` / `self.mode` 混用
3. Ablation 测试中 `daily_ev_loads` / `daily_battery_powers` 未初始化
4. `__pycache__` 缓存导致旧代码反复生效
5. 储能原始功率最初用 SOC 反推，后来改成直接记录
6. 储能 fallback 错误放在 `action == 2` 分支里，后续修正
7. 电价函数最终简化为固定分时电价

这些问题说明当前项目代码已经具有较强工程演化痕迹，后续如果继续深入优化，建议逐步清理结构。

---

## 11. 当前最值得继续做的事情

### 第一优先级：空调逻辑重构

建议像 EV 一样，把空调逐步改成结构化形式，例如：

```text
P_ac,target = P_base + Delta_P_pv + Delta_P_load + Delta_P_comfort
```

让空调也从：

- 固定绝对阈值
- 多层启发式 if-else

过渡到：

- 相对光伏
- 相对净负荷
- 舒适度修正
- 统一功率表达

### 第二优先级：储能从 fallback 过渡到主策略主动放电

当前 fallback 已经把储能行为救回一部分，后续需要进一步让：

- `action == 0`
- 主策略自身

更稳定地学会放电。

### 第三优先级：EV 增加更多成本敏感性

当前 EV 已经过于强充电、强吸纳，后续若想进一步压成本，可能要让 EV 在中午内部更分层，而不是一味长时间高功率充电。

## 12. 一句话总总结

**本轮对话的核心成果，是把项目中最关键的 EV 与电价逻辑系统地相对化、结构化，并初步把储能从“几乎只充不放”拉回到“部分日型可放电”的状态；与此同时，也清楚暴露出空调模块仍然停留在大量绝对阈值与强启发式驱动的旧式工程逻辑阶段，是当前三类资源中最不统一、也最值得继续深入重构的部分。**
