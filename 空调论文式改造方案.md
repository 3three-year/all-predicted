# 空调论文式改造方案

## 一、背景与目标

当前项目中，空调对应柔性负荷动作 `action == 2`。现有实现能够参与光伏消纳和削峰，但本质上属于**工程化的直接功率控制模型**，即根据当前光伏、负荷和时间段，直接给空调目标功率 `target_power`。

这种方式存在两个问题：

1. 不够符合真实空调控制机理  
   现实中通常是通过调节设定温度、风量或运行模式来间接影响功率，而不是直接指定空调功率。

2. 不利于论文表达  
   当前逻辑中保留了大量绝对阈值和经验规则，结构不够统一，也不便于和 EV、储能等模块形成一致的建模语言。

因此，建议将空调从“直接调功率的柔性负荷”改造为：

**通过调节设定温度 `T_set`，再由室内温度偏差和设定温度变化共同决定空调目标功率的论文式模型。**

---

## 二、当前空调模型现状

### 1. 当前模型本体

当前空调类为：

- `AirConditioningLoad`
- 继承自 `FlexibleLoad`

现有重要状态和参数包括：

- `max_power`
- `min_power`
- `response_time`
- `ramp_rate`
- `current_power`
- `target_power`
- `current_temp`
- `comfort_temp`

### 2. 当前功率更新方式

当前空调真实功率并不会一步跳到目标功率，而是通过爬坡和平滑逐步更新。核心关系可写成：

```text
power_diff = target_power - current_power
max_delta = ramp_rate * delta_time / response_time
delta = clip(power_diff, -max_delta, max_delta)
current_power = current_power + delta
```

这部分可以保留，因为它符合设备存在响应惯性的事实。

### 3. 当前温度更新方式

当前空调已经有一个简化室温模型，可近似写成：

```text
T_in(t+1)
= T_in(t)
+ [ a_out * (T_out(t) - T_in(t))
   - a_ac * (P_ac(t) / P_ac,max)
  ] * Δt
```

目前代码中对应的量级大致是：

- `a_out ≈ 0.05`
- `a_ac ≈ 0.10`

并且有室温边界：

```text
16°C <= T_in <= 30°C
```

这部分也可以保留。

### 4. 当前动作调控方式

当前 `action == 2` 逻辑本质是：

```text
看 pv_output / current_load / 时段 / 算法类型
-> 直接构造 target_power
```

典型规则包括：

- `pv_output > 3.0 / 4.0`
- `pv_output < 1.0 / 2.0`
- `current_load > 0.2 / 0.3 / 0.4 / 0.5 / 1.0 / 1.2 / 1.5 / 2.0`

并直接给出：

- 正功率：高光伏时增加空调功率吸纳电能
- 负功率：高负荷时削减空调负荷参与削峰

---

## 三、当前模型的主要问题

### 1. 直接调功率，不够真实

现实中空调通常不是直接接收一个“目标功率命令”，而是通过：

- 调节设定温度
- 调节风速/档位
- 调节模式

从而在热平衡下改变功率。

因此，当前“直接设功率”更像：

**电力系统对可调负荷做的抽象控制**

而不是：

**真实空调温控设备的控制机理**

### 2. 允许负功率不够物理

当前空调设置里允许：

```text
min_power < 0
```

这可以解释为“相对基准空调负荷的削减”，但如果直接把它看作设备功率，就不物理，因为真实空调功率不可能小于 0。

### 3. 绝对阈值过多

例如：

- `pv_output > 3.0`
- `pv_output > 4.0`
- `pv_output < 1.0`
- `current_load > 1.2`
- `current_load > 1.5`

这些固定绝对值：

- 不适应不同天气/季节/倍率
- 和 EV/储能当前使用的相对量逻辑不一致
- 论文里不够好解释

### 4. 中午可能与 EV、储能重复吸纳

当前中午高光伏场景下，系统里可能同时发生：

- EV 高功率充电
- 储能充电
- 空调提高功率吸纳

这会导致：

- 光伏消纳率上升
- 但中午总负荷被抬高
- 成本不一定最优

### 5. 启发式依赖较强

动作选择层中，大量启发式会偏向：

```text
[2, 3, 4]
```

其中 `2` 就是空调动作。  
因此，空调调节并不完全是模型自己学出来的，存在明显的人工规则引导成分。

---

## 四、建议的新模型思路

### 核心原则

将当前空调动作从：

```text
直接构造目标功率
```

改成：

```text
动作先改变设定温度 T_set
-> 再根据 T_in - T_set 和 ΔT_set 计算目标功率
-> 再做功率约束和平滑
```

即：

**空调动作不再直接控制功率，而是控制设定温度。**

---

## 五、建议保留的部分

以下内容建议继续保留：

1. `AirConditioningLoad` 类本体
2. `current_temp` 作为室内温度 `T_in`
3. 功率爬坡和平滑更新逻辑：
   - `set_target()`
   - `update()`
   - `response_time`
   - `ramp_rate`
4. 当前简化温度动态模型

这样可以避免一次性推翻整个空调模块。

---

## 六、建议新增或明确的状态量

建议将空调相关状态显式整理为：

- `T_in`
  室内温度
- `T_set`
  设定温度
- `T_out`
  室外温度
- `P_ac`
  当前实际空调功率
- `P_ac_target`
  当前目标空调功率

### 建议初值

- `T_set_init = 26°C`
- `T_in_init = 25~26°C`

其中：

- 当前代码里的 `current_temp` 可直接视为 `T_in`
- 当前代码里的 `comfort_temp` 可作为 `T_set_init`

---

## 七、动作含义的重定义

### 当前含义

`action == 2` 表示：

**直接调节空调功率**

### 建议新含义

`action == 2` 表示：

**调高空调设定温度，以降低空调制冷功率**

第一版建议最简单处理：

- `action != 2`
  - `T_set` 保持不变
- `action == 2`
  - `T_set = T_set + ΔT_set`

建议：

```text
ΔT_set = +1°C
```

并限制设定温度范围：

```text
24°C <= T_set <= 28°C
```

---

## 八、目标功率计算公式

建议直接采用论文思路：

```text
P_ac,target(t)
= P_ac(t)
- k_set * P_ac,max * ΔT_set
+ k_in  * P_ac,max * (T_in(t) - T_set(t))
```

并限制：

```text
0 <= P_ac,target(t) <= P_ac,max
```

### 参数建议

先按论文值：

- `k_set = 0.0638`
- `k_in = 0.0627`

### 含义解释

#### 第一项 `P_ac(t)`
表示当前空调原本功率。

#### 第二项 `- k_set * P_ac,max * ΔT_set`
表示设定温度调高会直接降低空调功率。

#### 第三项 `+ k_in * P_ac,max * (T_in - T_set)`
表示如果室内温度仍高于设定温度，空调要恢复一部分功率来维持舒适。

---

## 九、温度动态模型建议

当前简化模型可继续沿用并规范表达：

```text
T_in(t+1)
= T_in(t)
+ [ a_out * (T_out(t) - T_in(t))
   - a_ac * (P_ac(t) / P_ac,max)
  ] * Δt
```

建议初始参数仍使用当前量级：

- `a_out = 0.05`
- `a_ac = 0.10`

并保留：

```text
16°C <= T_in <= 30°C
```

---

## 十、功率物理边界建议

建议真实空调功率今后统一为：

```text
0 <= P_ac <= P_ac,max
```

### 原因

真实空调设备功率不可能小于 0。  
如果要表示“削减负荷”，更合理的表述应该是：

**相对于基准空调功率减少了多少**

而不是直接让设备功率为负值。

这一步是提升物理一致性的关键改动。

---

## 十一、奖励设计建议

如果空调改成设定温度驱动，奖励也建议配套调整。

### 1. 舒适度惩罚

建议引入：

```text
R_comfort
= - w_temp * max(0, |T_in - T_comfort| - deadband)
```

建议：

- `T_comfort = 26°C`
- `deadband = 1°C`

含义：

- 25~27℃ 基本不罚
- 超出舒适带再开始罚

---

### 2. 高光伏时适度吸纳奖励

不建议再使用绝对 `pv_output` 阈值，建议用：

```text
pv_ratio = pv_output / day_pv_peak
```

当：

- `pv_ratio` 较高
- 且 `T_in > T_set`

可给一个温和奖励，表示：

**空调在不明显损害舒适度的前提下吸纳光伏**

---

### 3. 高净负荷时温度调节削峰奖励

当：

- `net_load_ratio` 高
- 执行了 `action == 2`
- `T_set` 被调高

可给一点削峰奖励，表示空调参与系统负荷削减。

---

## 十二、建议删除或逐步废弃的旧逻辑

如果按新方案改造，建议删除或逐步废弃下面这些空调主逻辑：

### 1. 删除空调动作主分支中的绝对光伏阈值

- `pv_output > 3.0`
- `pv_output > 4.0`
- `pv_output < 1.0`
- `pv_output < 2.0`

### 2. 删除空调动作主分支中的绝对负荷阈值

- `current_load > 0.2`
- `current_load > 0.3`
- `current_load > 0.4`
- `current_load > 0.5`
- `current_load > 1.0`
- `current_load > 1.2`
- `current_load > 1.5`
- `current_load > 2.0`

### 3. 删除“直接 target_power = 正/负某个值”的主赋值逻辑

保留：

- 功率限幅
- 平滑更新
- 温度更新

但不再保留经验式的主目标功率生成规则。

---

## 十三、建议的新执行流程

新的 `action == 2` 空调逻辑可按如下流程实现：

1. 读取当前状态：
   - `T_in`
   - `T_set`
   - `T_out`
   - `P_ac`
   - `pv_ratio`
   - `net_load_ratio`

2. 根据动作更新设定温度：
   - `action == 2` 时 `T_set = T_set + 1°C`
   - 否则 `T_set` 不变

3. 用论文公式计算目标功率：

   ```text
   P_ac,target
   = P_ac
   - k_set * P_ac,max * ΔT_set
   + k_in  * P_ac,max * (T_in - T_set)
   ```

4. 约束：

   ```text
   0 <= P_ac,target <= P_ac,max
   ```

5. 把 `P_ac,target` 交给现有平滑层：
   - `set_target()`
   - `update()`

6. 更新室内温度 `T_in`

7. 计算：
   - 舒适度惩罚
   - 高光伏温和吸纳奖励
   - 高净负荷削峰奖励

---

## 十四、和当前项目整体的关系

这套方案不会推翻项目现有结构，而是：

- 保留空调类本身
- 保留温度状态
- 保留功率平滑和爬坡约束
- 只重写 `action == 2` 的主调控逻辑

因此它属于：

**中等改动、但意义很大的结构化重构。**

---

## 十五、建议实施顺序

### 第一步
只改：

- `action == 2` 的主调控逻辑
- 动作先调 `T_set`
- 按论文公式生成 `P_ac_target`
- 保留现有平滑更新

### 第二步
再改：

- 奖励项重构
- 用 `pv_ratio` / `net_load_ratio` 逐步替代旧绝对阈值
- 彻底删除旧功率直设逻辑

---

## 十六、结论

论文中的空调调节核心是：

**通过调整设定温度，间接改变空调功率；目标功率由设定温度变化和室内温度偏差共同决定。**

而当前项目中的空调调节逻辑属于：

**基于光伏、负荷和时段经验规则直接设功率的柔性负荷工程抽象模型。**

因此：

- 用于电力系统调度仿真：可以成立
- 若作为“真实空调控制机理模型”：不够准确

更合理的改造方向是：

**保留现有功率平滑和温度动态框架，但把 `action == 2` 重构为“先调设定温度，再由论文公式计算目标功率”的设定温度驱动模型。**
