## 2. 项目中的主要资源与角色

当前项目中主要涉及三类柔性资源：

### 2.1 储能（Battery）

作用包括：

- 谷段充电、峰段放电
- 白天吸纳光伏
- 晚高峰或高净负荷时放电削峰
- 作为时移资源参与成本优化

### 2.2 EV（电动车）

作用包括：

- 完成用户给定充电任务
- 高光伏时吸纳新能源
- 在一定条件下可参与放电削峰

### 2.3 空调（Air Conditioning）

作用包括：

- 作为可调节柔性负荷参与光伏吸纳
- 高负荷时削减功率参与削峰
- 同时受简化温度动态和舒适度约束

## 3. 电价模型讨论与修改

### 3.1 原始电价设计

最初项目中的电价不是固定的峰谷平电价，而是状态反馈型动态价格。其结构大致为：

```text
price = base_price
      * seasonal_factor
      * day_factor
      * load_factor
      * pv_factor
      * soc_factor
```

其中：

- `seasonal_factor`：季节因子
- `day_factor`：工作日/周末因子
- `load_factor`：当前负荷因子
- `pv_factor`：光伏消纳相关因子
- `soc_factor`：储能 SOC 因子

### 3.2 为什么认为这套设计不合适

主要问题是：

1. 算法调度结果反过来改变算法自己面对的价格环境
2. 成本比较不再是“相同价格环境下的调度能力比较”
3. 与论文/答辩中的公平性要求不匹配

其中最不必要的是：

- `day_factor`
- `load_factor`
- `pv_factor`
- `soc_factor`

季节因子 `seasonal_factor` 还勉强可以解释为外生环境差异，但如果目标是纯算法对比，也可去掉。

### 3.3 最终修改后的电价方案

后来电价函数被整理成固定分时电价：

- 峰时：`8-11`, `18-22`
- 谷时：`23-7`
- 平时：其余时间取峰谷均值

即：

```text
price = base_price
```

### 3.4 当前成本计算方式

典型日运行成本统一按：

```text
Cost = Σ [ max(0, load(t) - pv(t)) * price(t) * 0.25 ]
```

其中：

- `load(t)`：算法调度后的总负荷
- `pv(t)`：当前光伏出力
- `price(t)`：固定分时电价
- `0.25`：15 分钟步长对应的小时数

该成本是**购电成本**，不包含卖电收益。

## 4. EV 逻辑演化总结

### 4.1 最初 EV 放电逻辑

早期 EV 放电触发逻辑主要是：

```text
pv_output < 1.0
current_load > 0.8
ev_load.soc > 0.4
```

然后再根据：

```text
peak_reduction = min(discharge_limit, (current_load - 0.8) * 0.5)
target_power = target_power - peak_reduction
```

### 4.2 旧逻辑问题

存在以下问题：

1. `pv_output < 1.0` 是固定绝对阈值
2. `current_load > 0.8` 也是固定绝对阈值
3. 不同天气、季节和光伏倍率下不稳定
4. 难以从论文角度解释为“净负荷削峰”

### 4.3 新 EV 放电逻辑：净负荷相对判断

后来改成了基于净负荷的连续判断：

定义：

```text
L_net = current_load - pv_output
L_base = day_mean_net_load
L_peak = day_peak_net_load
```

触发条件：

```text
L_net > (1 + eta_L) * L_base
且 SOC > 0.50
```

放电补偿量：

```text
Delta_P_peak = - min(discharge_limit, k_discharge * |P_ev_min| * net_load_ratio)
```

其中：

```text
net_load_ratio = clip((L_net - L_base) / (L_peak - L_base), 0, 1)
```

参数建议与讨论中形成的第一版是：

- `eta_L = 0.10`
- `k_discharge = 0.50`
- `SOC_dis_threshold = 0.50`

### 4.4 EV 充电逻辑：从绝对光伏值到相对 `pv_ratio`

原始场景划分为：

- 高光伏：`pv_output >= 3.0`
- 中光伏：`1.0 <= pv_output < 3.0`
- 低光伏：`pv_output < 1.0`

后续改成按当天峰值比例判断：

```text
day_pv_peak = max(day_pv_profile)
pv_ratio = pv_output / max(day_pv_peak, 1e-6)
```

并划分为：

- 高光伏：`pv_ratio >= 0.50`
- 中光伏：`0.15 <= pv_ratio < 0.50`
- 低光伏：`pv_ratio < 0.15`

这样可以避免固定绝对阈值在不同日型下漂移。

### 4.5 EV 奖励层统一

后来讨论并修改的方向是：

- 高光伏奖励不再用 `pv_output > 5.0`
- 低光伏惩罚不再用 `pv_output < 1.0 / 1.5`
- 统一改成按 `reward_pv_ratio` 判断

这使得：

- 决策层按 `pv_ratio`
- 奖励层也按 `pv_ratio`

口径统一。

### 4.6 EV 放电 SOC 阈值统一

早期代码和文档中曾存在：

- `0.4`
- `0.5`

两种口径。后续统一成：

```text
SOC > 0.50
```

### 4.7 EV 目标功率半结构化公式

在讨论中，EV 功率形成逻辑从“多层 if-else 场景规则”逐步抽象为：

```text
P_ev_target = P_task + Delta_P_pv + Delta_P_prog + Delta_P_peak
```

各项含义为：

- `P_task`
  当前任务底线功率，由 `required_power` 决定
- `Delta_P_pv`
  光伏支持项，光伏好时提高充电功率
- `Delta_P_prog`
  进度补偿项，任务落后时增强充电
- `Delta_P_peak`
  削峰项，净负荷高时向放电方向修正

这一步是论文表达上的重要统一。

### 4.8 EV fallback

后来又把 EV fallback 做成显式开关：

```text
enable_ev_fallback = False
```

含义：

- 默认关闭，用于研究公平对比
- 可对 DDQN / PPO 测试阶段打开，起到工程保底作用

### 4.9 当前 EV 存在的问题

从结果看，当前 EV 是三类资源里最完整的一块，但还有几个问题：

1. 白天中段充电偏强，可能抬高中午负荷
2. DDQN/PPO 的 EV 曲线包含 fallback 成分，不完全是纯策略结果
3. EV 放电削峰作用相比其充电吸纳作用仍不够突出

## 5. 储能逻辑演化总结

### 5.1 早期储能功率图问题：SOC 反推

最初储能功率图不是直接记录原始功率，而是用：

```text
P_bat(t) = [SOC(t) - SOC(t-1)] * E_bat / Δt
```

由 SOC 差分反推得到。

这种方法在趋势解释上是合理的，但存在问题：

1. 会放大 SOC 跳变
2. 不能直接代表环境中的原始控制命令
3. 跨天首点处理不自然
4. 更像“等效功率”而不是“原始功率”

### 5.2 后续修改：记录原始 `battery_power_kw`

后来储能功率改成在环境里每步直接记录：

```text
self.battery_power_kw
```

并由 trainer 在 `env.step(action)` 后直接采集，绘制 07a/07b 图。

这样图就变成了真正的原始储能功率曲线。

### 5.3 储能为什么会“只充不放”

在多轮分析中，我们明确了几个根因：

1. 充电路径很多：
   - 光伏弃光自动入储
   - `action == 1` 光伏充电
   - `action == 1` 谷段电网充电
2. 放电主入口几乎只有：
   - `action == 0`
3. T-DDQN / Ablation 启发式高度偏向 `[2,3,4]`
4. DDQN / PPO 没有储能放电 fallback

所以储能天然容易：

- SOC 上升
- 但很难下降

### 5.4 储能放电增强方案的讨论

曾讨论的最小增强方案包括：

1. 相对净负荷放电触发
2. 放电奖励
3. 高 SOC 不放电惩罚

后续又进一步讨论到：

- 把动作层 `action == 0` 概率释放出来
- 对 T-DDQN / Ablation 的启发式动作做定向修正
- 加环境层储能放电 fallback

### 5.5 后续发现的关键结构问题

在检查代码和图之后，发现一个很关键的问题：

储能 fallback 曾一度被错误地写进了 `action == 2` 分支，这导致：

- 并不是所有 `action != 0` 的时刻都能触发保底释放
- 而是只有某些动作场景才能生效

这个结构性问题比简单阈值更关键。

### 5.6 门槛放宽讨论

围绕下面这些门槛，进行了多轮讨论：

- `SOC > 0.70`
- `SOC > 0.55`
- 最终认为 `SOC > 0.40` 更适合当前四算法的 SOC 分布

同时还讨论了：

- `battery_net_load_ratio > 0.50`
- `price_ratio > 0.60`

这些条件是否过严，尤其是在“光伏已经下降、净负荷已高但还未到真正峰价”的时段。

### 5.7 当前储能结果状态总结

根据最新结果：

- 储能已经从“几乎完全不放电”改善成“部分典型日开始晚间释放”
- `T-DDQN / PPO / DDQN` 在部分日子里开始出现晚间 SOC 下降
- `07b` 图里已经可以看到更多负功率

但问题仍然存在：

1. 典型日 1 仍偏“重充轻放”
2. 放电对日型敏感
3. 仍较依赖 fallback 机制
4. Ablation 的储能几乎不起作用

## 6. 启发式动作选择的讨论

### 6.1 什么叫“启发式动作选择”

项目里的 T-DDQN / Ablation 并不是纯粹地由网络自己根据 Q 值或策略分布选动作，而是在动作层加了很多人为规则：

- 看光伏高不高
- 看负荷高不高
- 看时间段
- 再高概率直接返回 `[2,3,4]`

这种人为规则驱动的动作选择，就叫“启发式动作选择”。

### 6.2 启发式的作用

优点：

- 训练更容易出行为
- 柔性负荷更快被调起来
- 演示上更容易出现“看起来合理”的结果

缺点：

- 不是纯 RL
- 会压掉某些动作，例如储能放电动作 `0`
- 会掩盖真实策略能力

### 6.3 结论

当前 T-DDQN 不是纯 RL，而是：

```text
强化学习 + 领域知识启发式动作选择
```

这个判断对后续论文描述非常重要。

## 7. 当前空调模型完整梳理

### 7.1 空调模型本体

当前空调对应的是：

- `AirConditioningLoad`
- 继承自 `FlexibleLoad`

它本质上是一个**可调柔性负荷模型**，而不是严格的真实空调温控机理模型。

### 7.2 当前空调功率更新机制

空调的目标功率会先在：

```text
[min_power, max_power]
```

之间裁剪，然后再按响应时间和爬坡速率逐步更新真实功率。

功率更新可近似理解为：

```text
P_ac(t+1) = P_ac(t) + clip(P_target - P_ac(t), -max_delta, max_delta)
```

### 7.3 当前简化温度模型

当前空调还有一个简化温度动态，大致形式为：

```text
T_in(t+1)
= T_in(t)
 [ a_out * (T_out - T_in)
  - a_ac * (P_ac / P_ac,max)
 ] * Δt
```

含义：

- 空调功率越大，室温越往下拉
- 室外越热，室内越容易升温

### 7.4 当前空调参数

核心参数大致为：

- `max_power = 15.0`
- `min_power = -10.0`
- `response_time = 3`
- `ramp_rate = 2.0`
- `comfort_temp = 26.0`

其中 `min_power < 0` 不是指真实空调发电，而是表示：

**相对于基准空调负荷允许削减的柔性调节能力。**

### 7.5 当前动作 `action == 2` 的空调调控

空调动作的主逻辑不是“调设定温度”，而是：

- 看 `pv_output`
- 看 `current_load`
- 看时段
- 再直接给空调目标功率 `target_power`

也就是说，当前空调其实是在做：

**光伏吸纳与削峰型功率控制**

而不是论文中更真实的“设定温度调节型控制”。

### 7.6 当前空调残留的旧绝对阈值

空调逻辑中仍大量使用：

- `pv_output > 3.0`
- `pv_output > 4.0`
- `pv_output < 1.0`
- `pv_output < 2.0`

以及：

- `current_load > 0.2 / 0.3 / 0.4 / 0.5 / 1.0 / 1.2 / 1.5 / 2.0`

这些都是固定绝对阈值。

### 7.7 当前空调启发式问题

空调既在分支内部依赖阈值，又在动作层被启发式高概率推向：

```text
[2, 3, 4]
```

因此：

- 空调行为里有很强的人为规则成分
- 不像 EV 那样已经基本结构化

### 7.8 论文中的空调调节是什么意思

后来专门讨论了论文截图里的空调模型。

论文方法的核心不是“直接调功率”，而是：

1. 先调整空调设定温度 `T_set`
2. 再用下面的公式计算目标功率：

```text
P_ac,target
= P_ac
 - k_set * P_ac,max * ΔT_set
 + k_in  * P_ac,max * (T_in - T_set)
```

其中：

- `k_set = 0.0638`
- `k_in = 0.0627`

### 7.9 为什么说当前空调“不太符合实际”

因为现实里更常见的是：

- 改设定温度
- 改温控范围
- 让设备自己改变功率

而不是直接命令：

```text
空调当前吃多少 kW
```

所以当前空调模型更适合被表述为：

**电力系统中的柔性空调负荷抽象模型**

而不是：

**真实空调热工机理模型**

### 7.10 后续建议方向

建议未来把空调重构成：

- 动作先改 `T_set`
- 再按论文公式算 `P_ac,target`
- 继续保留当前功率平滑和温度动态

这样既不推翻项目，又能让空调逻辑更接近论文与实际。

## 8. 结合当前结果，对三类资源的系统判断

### 8.1 储能

当前状态：

- 已经从“完全不会放电”进步到“部分典型日可晚间释放”
- 但仍不稳定
- 典型日 1 仍偏弱
- Ablation 基本不用储能

结论：

**储能问题已经从“完全失效”转变成“会放但还不稳定，且较依赖 fallback”。**

### 8.2 EV

当前状态：

- 是三类资源中逻辑最完整的一块
- 已经比较稳定地参与白天调度
- 高光伏吸纳效果明显

问题：

- 中午充电偏强，成本压力仍较大
- DDQN/PPO 的 EV 曲线含 fallback 成分
- EV 放电削峰作用还不够显著

结论：

**EV 是当前最成熟的一类柔性资源，但偏“充电主导”，成本敏感性还不够细。**

### 8.3 空调

当前状态：

- 仍是最老、最工程化的一块
- 规则仍大量依赖绝对阈值和启发式
- 缺乏与 EV / 储能统一的结构化表达

结论：

**空调是当前最不规整、也最值得下一步继续系统重构的模块。**

### 9.1 成本图

- 有些算法虽然光伏吸纳率高，但成本并不最低

**是不是吸纳得太堆叠、太不经济。**

### 9.3 削峰填谷图

削峰效果已经有，但并不总能稳定转化为最优成本。  
这说明：

- 储能释放时机
- EV 充电强度
- 空调调节方式

三者还没有形成最优协同。

### 9.4 EV 图

EV 曲线已经很完整，说明 EV 是目前最积极的柔性资源。

### 9.5 储能 SOC 与功率图

这些图帮助识别出：

- 储能以前重充轻放
- 现在开始能放，但仍不稳定
- 对典型日差异较敏感

## 10. 本轮对话中的关键认知结论

1. 电价应改成固定分时电价，才能让成本比较更公平
2. EV 放电应该按净负荷相对超额程度判断，而不是固定绝对阈值
3. EV 充电应按 `pv_ratio` 相对光伏判断，而不是 `pv_output` 固定值
4. EV 奖励层应和 EV 主逻辑保持同一套相对量口径
5. 储能原始功率曲线应该直接记录，而不是由 SOC 差分反推
6. 储能从“会充不会放”到“部分会放”的过程，暴露出动作层与 fallback 结构的重要性
7. T-DDQN/Ablation 当前不是纯 RL，而是“RL + 启发式动作选择”
8. 空调当前是最不规整的一块，更像工程化柔性负荷抽象，而不是论文式设定温度模型

## 11. 后续建议工作顺序

基于本轮对话，我建议后续如果继续迭代，优先顺序如下：

### 第一优先级：空调逻辑重构

原因：

- 空调旧阈值最多
- 动作启发式依赖最强
- 与 EV / 储能中午可能重复吸纳
- 论文表达最弱

建议方向：

- 动作改为“设定温度调节”
- 目标功率按论文公式计算
- 再保留当前平滑和温度更新层

### 第二优先级：储能从 fallback 驱动过渡到主策略主动放电

原因：

- 现在放电开始能出现，但仍不稳定
- fallback 是救火，不是终局

### 第三优先级：给 EV 再加一点成本敏感性

原因：

- EV 当前偏强充电吸纳
- 对成本的抬升压力仍然存在

## 12. 一句话总总结

**本轮对话的核心，是把项目中三类柔性资源的调度逻辑逐步从“工程化规则堆叠”往“相对量判断、结构化表达、可论文解释”的方向推进：EV 已经基本完成主线整理，储能开始建立充放闭环但仍不稳定，空调则仍是最值得继续重构的一块。**
