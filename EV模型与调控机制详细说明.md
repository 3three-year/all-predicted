# 当前项目 EV 模型与调控机制详细说明

## 文档说明

本文档专门整理当前项目中所有与 EV（电动汽车柔性负荷）相关的建模、公式、任务层设计、充放电调控逻辑、不同算法下的控制差异，以及这些设计是否合理、哪些地方是工程化简化。  

本文档不再泛泛讲“EV 是柔性负荷”，而是尽量贴近当前代码实现，把以下几个问题讲清楚：

1. 当前项目中的 EV 到底是什么模型。
2. EV 的参数分别是什么意思。
3. EV 的 SOC 动态公式、任务层公式、满意度公式分别在表达什么。
4. EV 在动作 3 下到底是怎么形成目标功率的。
5. EV 是什么时候充电、什么时候放电、什么时候不动作。
6. T-DDQN、DDQN、PPO、Baseline 对 EV 的调控有什么差异。
7. 这些设计是否合理，哪些地方是工程近似，哪些地方若写论文需要特别说明。

全文默认基于当前项目代码实现理解，核心环境逻辑主要来自 `rural_env.py`。

---

## 1. EV 在当前项目中的定位

先说最核心的一句话：

**当前项目中的 EV，不是普通固定负荷，也不是完全自由的电池储能，而是“带充电任务约束的聚合柔性负荷”。**

这句话决定了后面所有公式和调控逻辑应该如何理解。

### 1.1 EV 的三重身份

当前项目中的 EV 同时具有三层属性：

1. 负荷属性  
   当 EV 充电时，会增加台区当前负荷。

2. 储能属性  
   在一定条件下，EV 可以放电，参与削峰。

3. 任务属性  
   EV 不是想怎么调就怎么调，它还必须完成“当天应该充入多少电”的任务。

所以项目对 EV 的根本处理方式不是：

```text
把 EV 当成一个可随意充放的自由电池
```

而是：

```text
在保证 EV 基本充电任务完成的前提下，
尽可能利用 EV 参与光伏消纳、削峰填谷和成本优化
```

### 1.2 当前项目更接近聚合 EV 模型

当前项目中的 EV 不是单车精细行为模型，而是更接近：

**聚合 EV 柔性负荷模型**

原因如下：

1. 没有为每辆车单独建模到站时间、离站时间、出发需求。
2. 没有每辆车单独的目标 SOC 曲线。
3. 用的是参考 EV 负荷曲线和总任务量，不是单车时空轨迹。
4. 任务层关注的是“当天总共该充多少电”，而不是“某辆车几点前必须充到 80%”。

因此，后续所有公式都应理解为：

```text
针对聚合 EV 充电任务与聚合 EV 调度能力的建模
```

## 2. EV 相关参数及其物理意义

当前项目中 EV 相关主要参数如下。

### 2.1 最大充电功率 `P_ev_max = 20.0`

含义：

```text
EV 在任一时刻最多只能以 20 的功率充电
```

物理意义：

1. 反映充电桩/聚合充电资源的最大充电能力。
2. 防止控制策略给出不现实的大功率充电指令。

合理性分析：

1. 合理。
2. 任何真实 EV 充电系统都存在最大充电功率边界。
3. 这是最基本的设备能力约束。

---

### 2.2 最大放电功率下界 `P_ev_min = -15.0`

当前项目中约定：

- `P_ev(t) > 0` 表示充电
- `P_ev(t) < 0` 表示放电

因此：

```text
P_ev_min = -15.0
```

表示：

```text
EV 最多只能以 15 的功率向外放电
```

物理意义：

1. EV 虽然可参与 V2G/削峰，但放电能力有限。
2. 放电功率通常比充电功率更保守。

合理性分析：

1. 合理。
2. 充放电功率不对称在工程上很常见。
3. 放电更保守也更符合用户接受度和设备保护逻辑。

---

### 2.3 响应时间 `T_ev_resp = 8`

含义：

```text
EV 的功率切换不是瞬时完成，而是需要一个响应过程
```

物理意义：

1. 目标功率从当前值变化到新值，不是一步跳变，而是渐进逼近。
2. EV 的动态变化比空调更慢，体现其调节惯性更强。

合理性分析：

1. 合理。
2. 真实 EV 调节涉及充电设备控制、电池管理系统、充放电切换保护。
3. 不可能无延迟、无惯性瞬时切换。

---

### 2.4 爬坡速率 `R_ev_ramp = 1.5`

含义：

```text
EV 功率每一步最多能变化多少
```

物理意义：

1. 与响应时间共同决定功率变化快慢。
2. 防止相邻时段功率大幅跳变。

合理性分析：

1. 合理。
2. 真实设备均存在功率变化速率限制。
3. 对强化学习而言，这也有利于训练稳定性。

---

### 2.5 初始 SOC `SOC_ev(0) = 0.6`

含义：

```text
EV 初始电量为 60%
```

物理意义：

1. 既不是接近空电，也不是接近满电。
2. 给系统保留了充电空间与放电空间。

合理性分析：

1. 合理。
2. 作为聚合 EV 的初始状态，这是一个较自然的中间值。

---

### 2.6 电池容量 `E_ev = 200.0`

含义：

```text
EV 聚合电池容量为 200
```

在当前模型中，容量的主要作用是：

```text
把功率变化换算成能量变化，
再把能量变化换算成 SOC 变化
```

合理性分析：

1. 从聚合 EV 模型角度合理。
2. 如果按单车理解，这个数会偏大，因此必须强调它更像“聚合资源等效容量”。

---

### 2.7 放电允许阈值 `SOC_ev_dis = 0.5`

含义：

```text
只有当 SOC_ev(t) >= 0.5 时，EV 才允许放电
```

物理意义：

1. 不是只要电池还能放电就让它放。
2. 用户侧必须保留一定剩余电量，系统才允许调用 EV 放电能力。

合理性分析：

1. 非常合理。
2. 体现了 EV 是用户资产，不是完全自由的系统储能。

注意事项：

当前代码已经统一为 `SOC_ev(t) > 0.5` 的放电触发口径。  
如果后续写论文，可以直接将其表述为“EV 仅在荷电状态高于 50% 时参与放电削峰”，不再需要区分“理论放电阈值”和“具体削峰分支触发阈值”。

## 3. EV 的基础物理模型：SOC 动态

EV 的最基础建模是 SOC 随时间变化的动态。

### 3.1 充电时 SOC 更新公式

公式：

```text
SOC_ev(t+1) = SOC_ev(t) + P_ev(t) * delta_t / E_ev
```

条件：

```text
P_ev(t) > 0
```

### 3.1.1 各部分含义

- `SOC_ev(t)`：当前时刻 EV SOC
- `P_ev(t)`：当前充电功率
- `delta_t`：时间步长，当前为 `0.25 h`
- `E_ev`：聚合 EV 等效容量

### 3.1.2 公式表达的物理含义

它表达的是：

```text
下一时刻 SOC
= 当前 SOC
+ 当前时段充进去的电量 / 总容量
```

其中：

- `P_ev(t) * delta_t` 是当前 15 分钟充进去的能量
- 再除以总容量，就得到 SOC 增量

### 3.1.3 是否合理

完全合理。  
这是标准电池能量守恒建模。

---

### 3.2 放电时 SOC 更新公式

公式：

```text
SOC_ev(t+1) = SOC_ev(t) - abs(P_ev(t)) * delta_t / E_ev
```

条件：

```text
P_ev(t) < 0
```

### 3.2.1 各部分含义

- `abs(P_ev(t))`：放电功率幅值
- `abs(P_ev(t)) * delta_t`：本时段放出去的能量
- 除以 `E_ev`：换算成 SOC 降低量

### 3.2.2 公式表达的物理含义

```text
EV 放出去多少电，SOC 就下降多少
```

### 3.2.3 是否合理

方向上完全合理。  
但这里没有显式加入 EV 充放电效率，因此是一个工程简化版本。

更严格的写法通常应当区分：

1. 充电效率
2. 放电效率

当前项目为了简化 RL 环境，没有把 EV 效率单独展开。

---

### 3.3 SOC 边界约束

公式：

```text
0 <= SOC_ev(t) <= 1
```

含义：

1. `0` 表示空电
2. `1` 表示满电

合理性分析：

1. 必须有。
2. 没有这个约束会出现不可能状态。

---

## 4. EV 任务层：当前项目 EV 建模的核心

如果只看上面的 SOC 公式，你得到的只是一个“可充可放电池”。  
但当前项目真正的关键在于：它还给 EV 定义了“任务层”。

这个任务层解决的问题是：

```text
今天总共要为 EV 完成多少充电任务？
到当前时刻，这个任务该推进到什么程度？
```

---

### 4.1 参考总任务能量 `reference_energy`

当前代码中的 `reference_energy` 对应：

```text
E_ev_ref
```

表示：

```text
当天 EV 的参考总充电任务量
```

公式：

```text
E_ev_ref = sum( max(real_ev_load_day(k), 0) ) * 0.25
```

### 4.1.1 各部分含义

- `real_ev_load_day(k)`：第 `k` 个时段的参考 EV 负荷
- `max(real_ev_load_day(k), 0)`：只保留正向充电部分
- `sum(...)`：全天累加
- `0.25`：每个时间步长是 0.25 小时

### 4.1.2 这个公式的真实含义

它表示：

```text
当天参考 EV 负荷曲线中，原本应当充进去的总能量
```

也就是说，项目把当天真实 EV 负荷曲线抽象成一个：

```text
“当天必须完成的总充电任务”
```

### 4.1.3 是否合理

合理，特别适合聚合 EV 模型。  
它不是单车目标 SOC 建模，而是聚合任务量建模。

---

### 4.2 已完成能量 `actual_ev_energy_charged`

当前代码中的 `actual_ev_energy_charged` 对应：

```text
E_ev_done(t)
```

表示：

```text
截至当前时刻，系统已经累计为 EV 充进去多少电
```

更新逻辑可概括为：

```text
E_ev_done(t+1) = min( E_ev_ref, E_ev_done(t) + max(P_ev(t), 0) * delta_t )
```

### 4.2.1 为什么只统计 `max(P_ev(t), 0)`

因为当前项目把“任务完成量”定义为：

```text
累计正向补能量
```

这意味着：

1. 充电会增加已完成量
2. 放电不会把已完成量再扣回去

### 4.2.2 这表示什么

说明当前任务层更像：

```text
累计补能进度模型
```

而不是严格的净能量平衡模型。

### 4.2.3 是否合理

作为工程任务层合理，但要注意它是简化的：

1. 放电后不会回扣已完成任务量
2. 更像“今天给 EV 补了多少能量”而不是“最终净注入电量”

这在论文里最好明确为工程近似。

---

### 4.3 剩余任务能量 `remaining_energy`

当前代码中的 `remaining_energy` 对应：

```text
E_ev_remain(t)
```

定义：

```text
E_ev_remain(t) = max( 0, E_ev_ref - E_ev_done(t) )
```

含义：

```text
截至当前时刻，为了完成今天 EV 任务，还差多少电量
```

合理性分析：

1. 非常合理。
2. 它是后续计算“任务紧不紧”的基础变量。

---

## 5. EV 的阶段性进度机制

当前项目没有只在日末检查 EV 任务是否完成，而是引入了“阶段性进度目标”。

其本质是：

```text
不仅关心今天总任务量最终是否完成，
也关心这个任务是否按照合理节奏推进
```

---

### 5.1 期望完成比例 `expected_completion`

当前代码中，`expected_completion` 来自一个分段进度函数 `_expected_ev_completion(time_step)`。

其物理意义是：

```text
若当前已经走到一天中的某个时刻，
理论上 EV 充电任务应完成到多少比例
```

设 `t_step` 为当天第几个 15 分钟步长，则：

#### 阶段 1：0:00-6:00

```text
if t_step < 24:
    r_exp(t_step) = 0.0
```

含义：

```text
凌晨前 6 小时不要求完成 EV 任务
```

#### 阶段 2：6:00-12:00

```text
r_exp(t_step) = 0.3 * (t_step - 24) / 24
```

含义：

```text
从 6:00 到 12:00，期望进度从 0 线性上升到 30%
```

#### 阶段 3：12:00-18:00

```text
r_exp(t_step) = 0.3 + 0.45 * (t_step - 48) / 24
```

含义：

```text
从 12:00 到 18:00，期望进度从 30% 上升到 75%
```

#### 阶段 4：18:00-21:00

```text
r_exp(t_step) = 0.75 + 0.15 * (t_step - 72) / 12
```

含义：

```text
从 18:00 到 21:00，期望进度从 75% 上升到 90%
```

#### 阶段 5：21:00-24:00

```text
r_exp(t_step) = min(1.0, 0.9 + 0.1 * (t_step - 84) / 12)
```

含义：

```text
最后 3 小时完成剩余 10%
```

---

### 5.2 为什么分母会是 24 或 12

这是很多人第一次看会困惑的地方。

原因非常简单：

1. 一天有 96 个 15 分钟步长
2. 6 小时 = 24 个步长
3. 3 小时 = 12 个步长

所以：

```text
(t_step - 起点) / 24
```

表示：

```text
当前在这个 6 小时阶段内已经走了多少比例
```

这本质上是在做：

```text
分段线性进度插值
```

---

### 5.3 当前完成比例 `completion_ratio`

定义：

```text
completion_ratio = actual_ev_energy_charged / reference_energy
```

含义：

```text
到当前时刻，今天 EV 任务已经完成了多少比例
```

---

### 5.4 进度偏差 `slack` / `progress_gap`

当前代码中进度偏差可理解为：

```text
slack = expected_completion - completion_ratio
```

含义：

1. `slack > 0`：当前任务进度落后
2. `slack < 0`：当前任务进度超前

这一步的意义是：

```text
把“当前紧不紧”量化出来
```

---

## 6. EV 进度奖励与最终完成奖励

EV 在当前项目中有两类任务相关奖励：

1. 过程奖励：看每一步进度是否落后
2. 最终完成奖励：看总任务完成质量

---

### 6.1 过程层：进度奖励 `ev_progress_reward`

当当天有参考任务时：

```text
current_progress = actual_ev_energy_charged / reference_energy
expected_progress = expected_completion
progress_gap = expected_progress - current_progress
```

若落后：

```text
if progress_gap > 0:
    ev_progress_reward = -2.5 * progress_gap
```

若超前：

```text
else:
    ev_progress_reward = min(1.0, -progress_gap * 1.5)
```

含义：

1. 落后就惩罚
2. 略微超前有奖励
3. 超前奖励上限封顶为 `1.0`

设计原因：

1. 防止任务一整天不动，最后再补
2. 鼓励按合理节奏推进
3. 避免超前过多反而让策略一上来就猛充

合理性分析：

1. 合理。
2. 这是典型的“过程对齐型奖励”。
3. 系数 `2.5` 和 `1.5` 属于经验参数，是工程调参结果，不是理论常数。

---

### 6.2 结果层：任务完成奖励 `ev_energy_fulfillment_reward`

当前代码会计算完成度：

```text
fulfillment_ratio = actual_ev_energy_charged / reference_energy
```

然后按区间给奖励或惩罚。

#### 情况 1：完成度在 `[0.9, 1.1]`

```text
ev_energy_fulfillment_reward = 2.0
ev_energy_fulfillment_reward += 1.0 * (1.0 - abs(fulfillment_ratio - 1.0) / 0.1)
```

含义：

1. 只要在合理容忍区间内，就给基础奖励
2. 越接近 100%，额外奖励越高

#### 情况 2：完成度小于 `0.9`

```text
shortage = 0.9 - fulfillment_ratio
ev_energy_fulfillment_reward = -5.0 * shortage
```

若严重不足，即 `< 0.7`：

```text
ev_energy_fulfillment_reward -= 5.0 * (0.7 - fulfillment_ratio)
```

含义：

1. 充不够就惩罚
2. 严重充不够还加重惩罚

#### 情况 3：完成度大于 `1.1`

```text
excess = fulfillment_ratio - 1.1
ev_energy_fulfillment_reward = -2.0 * excess
```

含义：

1. 过充也不理想
2. 但过充惩罚轻于不足惩罚

合理性分析：

1. 合理。
2. “不足比过充更严重”非常符合现实。
3. 容忍区间 `[0.9, 1.1]` 是工程近似，不是物理定律。

## 7. EV 在综合用户满意度中的作用

当前项目没有单独定义一个复杂的 EV 满意度函数，而是把 EV 的可用性直接并入综合用户满意度。

综合满意度计算形式为：

```text
S_user(t) = w_th(t) * C_thermal(t) + w_ev(t) * SOC_ev(t)
```

其中：

- `C_thermal(t)`：空调热舒适度
- `SOC_ev(t)`：EV 当前 SOC
- `w_th(t)`：热舒适权重
- `w_ev(t)`：EV 可用性权重

含义：

1. EV 的电量越高，用户越有安全感
2. EV 的电量越低，用户满意度越低
3. EV 和空调一起构成用户侧综合满意度

合理性分析：

1. 合理，但属于简化模型。
2. 实际用户对 EV 电量的感知不一定线性。
3. 这里直接用 SOC 代表可用性，是一种工程上便于训练的代理建模。

---

## 8. EV 充放电行为机理：动作 3 下到底怎么调

这一部分是整个 EV 逻辑最关键的一层。

一定要先明确：

**当前项目里，EV 的实际充放电目标功率，不是由一条简单闭式公式一次算出来，而是由任务层、光伏层、进度层、负荷层和功率平滑约束共同形成。**

也就是说，它不是：

```text
target_power = A + B + C
```

而更接近：

```text
先算任务底线功率 required_power
再算光伏驱动项 pv_priority
再看当前进度是否落后 slack
然后按场景规则选出 target_power
最后如果处于削峰场景，再把 target_power 往负方向修正
```

下面按真实逻辑顺序讲。

---

### 8.1 先判断今天有没有充电任务

代码首先检查：

1. `reference_energy <= 1e-6`
2. `remaining_energy <= 1e-6`
3. `ev_available == False`

若任一条件成立，则：

```text
target_power = 0.0
```

#### 为什么要这样设计

因为当前项目里的 EV 首先是“有任务约束的用户资源”，不是完全自由的储能。

所以：

1. 没任务就不调
2. 任务已经完成就不再调
3. EV 不可用就不调

合理性分析：

1. 合理。
2. 这是保证 EV 不被无限制调用的第一道边界。

---

### 8.2 计算任务底线功率 `required_power`

公式：

```text
required_power = remaining_energy / (remaining_steps * 0.25)
```

其中：

- `remaining_energy`：今天还剩多少充电任务量（kWh）
- `remaining_steps`：从当前时刻到日末还剩多少个 15 分钟时间步
- `0.25`：每个时间步对应 0.25 小时

因此：

```text
remaining_steps * 0.25
```

表示：

```text
从现在到今天结束，还剩多少小时可以用来补任务
```

整个公式的物理意义是：

```text
如果把剩余任务平均分摊到后续所有时段，
当前至少应该对应多大的平均充电功率，
才能在今天结束前刚好完成任务
```

这不是最终功率，只是一个：

```text
任务底线功率
```

合理性分析：

1. 非常合理。
2. 它体现的是任务紧迫度，不只是任务总量。
3. 剩余任务一样多，但剩余时间越少，`required_power` 越大，这非常符合实际。

---

### 8.3 计算光伏优先充电项 `pv_priority`

代码里会算：

```text
pv_priority = min(max_positive_power, pv_output * 0.85)
```

某些 fallback 逻辑中用 `0.9`。

#### 各部分含义

- `pv_output`：当前时刻光伏出力
- `pv_output * 0.85`：从光伏吸纳角度出发，希望 EV 当前承担的优先充电功率
- `max_positive_power`：当前允许的最大正向充电功率

#### 为什么要和 `max_positive_power` 取最小值

因为即使当前光伏很高，EV 也不能：

1. 超过设备能力上限
2. 超过当前任务允许的最大正向充电空间

所以 `pv_priority` 表示的是：

```text
从光伏吸纳视角看，当前最希望 EV 充多少
```

而不是最终必然执行值。

#### 为什么乘 `0.85`

因为项目并不希望把全部光伏都硬分给 EV。  
系统里还有：

1. 基础负荷
2. 储能
3. 空调
4. 可平移负荷

所以这里乘一个系数，表示：

```text
光伏高时，EV 应优先多充，
但不一定独占全部光伏
```

合理性分析：

1. 合理。
2. 这是一种典型的光伏驱动项设计。
3. `0.85` 或 `0.9` 属于经验系数。

---

### 8.4 计算进度偏差 `slack`

代码中：

```text
slack = expected_completion - completion_ratio
```

其中：

- `expected_completion`：当前时间点理论应完成比例
- `completion_ratio`：当前实际完成比例

含义：

1. `slack > 0`：进度落后
2. `slack < 0`：进度超前

作用：

```text
slack` 用来告诉系统：
当前 EV 任务到底紧不紧
```

合理性分析：

1. 合理。
2. 它把任务层从“总量约束”扩展成了“过程进度约束”。

---

### 8.5 `required_power`、`pv_priority`、`slack` 如何共同形成 `target_power`

这一点最容易让人混乱，所以这里单独讲。

当前代码**不是线性相加**：

```text
target_power != required_power + pv_priority + slack
```

而是通过场景分支来形成 `target_power`。

#### 场景 A：高光伏 `pv_output >= 3.0`

代码思想：

```text
target_power = max(required_power, pv_priority)
```

含义：

1. 任务底线不能丢；
2. 既然当前光伏高，如果光伏驱动值更大，就应该抓住窗口多充一点。

也就是说：

```text
高光伏时，光伏项会把目标功率往上拉
```

#### 场景 B：中等光伏 `1.0 <= pv_output < 3.0`

如果 `slack > 0.05`，说明任务已有明显落后，则：

```text
target_power = max(required_power, pv_priority * 0.6)
```

含义：

1. 任务落后了，要加快充电；
2. 但光伏不算很强，所以光伏项打个折。

如果 `slack <= 0.05`，说明任务并不紧，则：

```text
target_power = max(0.0, required_power * 0.5)
```

含义：

```text
任务不紧，光伏也不算很高，
那就先保守一点，不必按 full required_power 去充
```

#### 场景 C：低光伏 `pv_output < 1.0`

如果：

- `slack > 0.1`，说明任务落后明显  
或  
- `current_time_step >= 84`，说明已接近日末

则：

```text
target_power = max(required_power, self.ev_load.max_power * 0.2)
```

含义：

```text
即使当前光伏低，也要开始兜底补任务
```

否则：

```text
target_power = 0.0
```

含义：

```text
当前光伏低，任务也不紧，
那就干脆先不充，等待更好的时间窗口
```

#### 这一整套设计的本质

可以把它总结为：

1. `required_power` 给出任务底线；
2. `pv_priority` 给出光伏驱动上拉；
3. `slack` 决定当前是否需要优先赶任务；
4. 三者不是直接线性相加，而是在不同光伏场景下通过 `max`、缩放、置零等方式共同形成目标功率。

---

### 8.6 EV 放电什么时候触发

EV 放电不是默认行为，而是在满足特定削峰场景时触发。

当前主要条件是：

```text
pv_output < 1.0
current_load > 0.8
ev_load.soc > 0.5
```

含义：

1. 光伏很低  
   当前不能指望光伏支撑系统。

2. 当前负荷高  
   台区存在明显削峰需求。

3. EV SOC 足够  
   EV 具备可释放电量的空间。

这表示：

```text
只有在低光伏、高负荷、SOC 较充足时，
EV 才从“补能资源”切换成“削峰资源”
```

---

### 8.7 放电功率怎么计算

先算放电能力上限：

```text
discharge_limit = min(
    self.ev_load.max_power,
    (self.ev_load.soc - 0.5) * self.ev_load.capacity / 0.25
)
```

#### 第一项：`self.ev_load.max_power`

表示设备能力上限。

#### 第二项：`(self.ev_load.soc - 0.5) * self.ev_load.capacity / 0.25`

表示：

```text
如果最低只允许放到 0.5，
从当前 SOC 到 0.5 之间的可释放能量，
换算成当前 15 分钟步长下的最大等效放电功率
```

这一步把 SOC 裕量转换成了放电能力上限。

随后计算当前削峰所需放电量：

```text
peak_reduction = min(
    discharge_limit,
    k_discharge * P_ev_dis_max * min( (L_net - L_base_net) / (L_peak_net - L_base_net), 1.0 )
)
```

这里：

- `(L_net - L_base_net) / (L_peak_net - L_base_net)` 表示当前净负荷相对当天基准净负荷的超额比例
- `k_discharge` 表示 EV 并不承担全部超额净负荷，而只承担其中一部分

然后将其作用到目标功率上：

```text
target_power = max(target_power - peak_reduction, self.ev_load.min_power)
```

这表示：

```text
先形成一个偏充电或中性的目标功率，
再根据削峰需求把它往负方向修正，
使 EV 转入放电削峰
```

合理性分析：

1. 合理。
2. EV 放电不是随便放，而是同时受设备上限、SOC 裕量和当前削峰需求约束。
3. 这比“满足条件就满功率放电”更稳妥，也更贴近工程控制。

---

### 8.8 放电之后，任务完成量会不会被扣回

当前代码中：

```text
delivered_energy = max(self.ev_load.current_power, 0.0) * 0.25
```

也就是说：

1. 只有正向充电功率才会累加到 `actual_ev_energy_charged`
2. 放电不会把 `actual_ev_energy_charged` 扣回

因此，当前项目里的任务层更像：

```text
累计补能任务模型
```

而不是严格的：

```text
净能量结余模型
```

合理性分析：

1. 这是一个工程简化。
2. 如果要做更严格的 V2G 任务建模，后续可以考虑把放电引起的任务回补也写入任务层。

---

## 9. 不同算法对 EV 的调控区别

当前项目里，Baseline、DDQN、PPO、T-DDQN 对 EV 的调控并不完全相同。

---

### 9.1 Baseline

Baseline 基本不做智能 EV 调控。

逻辑上更接近：

1. 如果有参考真实 EV 曲线，则按参考曲线运行；
2. 否则保持不调。

因此它更像：

```text
EV 不参与主动优化
```

作用：

1. 作为对照组；
2. 用来衡量智能调度相对于基准运行的改进。

合理性分析：

1. 合理。
2. Baseline 的存在是实验对比必须的。

---

### 9.2 DDQN

DDQN 会主动调 EV，但整体风格偏保守。

#### 9.2.1 它考虑哪些量

DDQN 会综合考虑：

1. `remaining_energy`
2. `remaining_steps`
3. `pv_output`
4. `slack`
5. `current_load`
6. `SOC_ev(t)`

#### 9.2.2 它对光伏的敏感度

代码中：

```text
algo_pv_bias = 0.9
```

表示：

```text
DDQN 会参考光伏，但不会特别激进地追逐光伏瞬时峰值
```

#### 9.2.3 它的 fallback 机制

当 `action != 3` 时，DDQN 仍可能自动触发 EV 补电：

1. 任务未完成；
2. EV 可用；
3. 光伏高，或已经接近日末且进度落后严重；

则会自动给 EV 一个补电目标功率。

这说明：

```text
即使当前动作不是 EV，
系统也不希望 EV 任务严重拖欠
```

#### 9.2.4 整体风格

可以总结为：

```text
主动调 EV，兼顾任务和光伏，但整体较稳妥
```

---

### 9.3 PPO

PPO 和 DDQN 的 EV 调控思想相近，但更平滑、更保守。

#### 9.3.1 光伏偏置更低

代码中：

```text
algo_pv_bias = 0.75
```

这表示：

```text
PPO 比 DDQN 更不容易因为瞬时光伏高点而明显抬高 EV 充电功率
```

#### 9.3.2 更偏向平稳完成任务

PPO 更倾向于：

1. 平滑补任务；
2. 不强烈追逐瞬时窗口；
3. 调控更温和。

#### 9.3.3 fallback 机制也存在

与 DDQN 类似，当 `action != 3` 时，也会根据任务滞后程度触发自动补电。

#### 9.3.4 整体风格

可以总结为：

```text
更偏向平稳补任务，不像 DDQN 那样更看重瞬时光伏窗口
```

---

### 9.4 T-DDQN

T-DDQN 是当前项目里 EV 调控最值得重点讲的一类。

#### 9.4.1 它仍然保留任务层和场景层逻辑

T-DDQN 同样考虑：

1. 剩余任务
2. 光伏出力
3. 进度偏差
4. 当前负荷
5. EV SOC

所以它不是脱离工程约束完全自由学，而是在同样任务边界下进行智能调控。

#### 9.4.2 它也允许在低光伏高负荷时放电削峰

因此，T-DDQN 的 EV 既能吸纳光伏，也能在削峰场景下转入放电。

#### 9.4.3 它的最大特点是更强的平滑控制

代码中典型设置为：

```text
alpha = 0.2
max_change = self.ev_load.max_power * 0.2
```

含义：

1. 当前功率会和上一步功率做更强平滑；
2. 单步变化幅度更小；
3. 不会突然猛充猛放。

#### 9.4.4 为什么这么设计

因为 T-DDQN 更强调：

```text
在任务约束下平滑、稳定地承担光伏吸纳和削峰任务
```

而不是追求瞬时最激烈的功率变化。

#### 9.4.5 整体风格

可以总结为：

```text
T-DDQN 下的 EV 调控更平缓、更稳定，更强调全过程协同
```

---

## 10. 当前项目 EV 设计整体是否合理

这一部分适合答辩时总结。

### 10.1 合理的地方

1. 把 EV 建成了“负荷 + 储能 + 任务”的三层模型；
2. 有明确 SOC 动态；
3. 有任务总量约束；
4. 有阶段性进度目标；
5. 有最终完成度奖励；
6. 有放电阈值和削峰触发条件；
7. 有功率平滑约束；
8. 能与光伏消纳和削峰填谷目标协同。

### 10.2 简化和不足

1. 当前更像聚合 EV 模型，不是单车精细行为模型；
2. 已完成能量不会因后续放电而回扣；
3. 没有显式 EV 充放电效率；
4. 没有电池老化成本；
5. 没有单车到站/离站时间窗；
6. 期望进度曲线是工程先验，不是用户统计行为拟合；
7. EV 放电阈值已经统一为 `SOC > 0.5`，文档和代码口径现已一致。

### 10.3 结论性评价

可以这样评价当前 EV 模型：

```text
它不是面向单车出行行为仿真的高保真模型，
而是面向台区协同调度的工程化聚合 EV 柔性负荷模型。
在强化学习场景中，这种建模方式具有较强的可解释性、可训练性和调控稳定性。
```

## 11. 适合答辩时的口径

如果老师问：

### “你这个项目里 EV 是怎么调控的？”

可以答：

```text
当前项目中的 EV 不是自由电池，而是带充电任务约束的聚合柔性负荷。
系统先根据参考 EV 负荷定义当天总充电任务量，再根据当前已完成任务量和剩余时间计算任务底线功率，
同时结合当前光伏出力、任务进度偏差和负荷水平形成 EV 的目标功率。
在光伏高发时段，EV 优先吸纳光伏充电；在低光伏、高负荷且 SOC 充足时，EV 可转入放电削峰。
不同算法的差异主要体现在光伏敏感度、进度补偿保守程度和功率平滑强度上，其中 T-DDQN 更强调全过程平滑协同。
```

### “强化学习在 EV 上到底起什么作用？”

可以答：

```text
规则层规定 EV 今天必须完成多少任务以及大致进度要求，
强化学习负责学习在满足这些边界条件的前提下，如何更优地安排 EV 在不同时间段的充放电功率，
从而兼顾光伏消纳、削峰填谷和运行成本优化。
```

## 2026-05 代码同步说明：当前 EV 主逻辑

本节用于同步 `rural_env.py` 中当前真实生效的 EV 调控逻辑。若本文档前文某些段落仍保留旧版本“按高光伏 / 中光伏 / 低光伏分支直接设定 `target_power`”的写法，应以本节说明为准。

### 1. 当前 EV 主动作 `action == 3` 的结构

当前代码已不再采用：

```text
if 高光伏:
    target_power = ...
elif 中光伏:
    target_power = ...
else:
    target_power = ...
```

这种按场景直接赋值目标功率的写法。

当前代码改为先构造原始目标功率：

```text
target_power_raw = P_task + Delta_P_pv + Delta_P_prog + Delta_P_peak
```

然后再对 `target_power_raw` 做：

1. 任务存在性与 EV 可用性门控；
2. 充放电功率上下限裁剪；
3. 各算法对应的平滑与单步变化约束。

因此，更准确的表述是：

```text
最终执行功率 != 四项简单裸相加
最终执行功率 = constrained_and_smoothed(target_power_raw)
```

### 2. 四项结构与代码对应

#### 2.1 任务项 `P_task`

当前代码中：

```text
required_power = remaining_energy / (remaining_steps * 0.25)
P_task = min(max_positive_power, required_power)
```

含义是：若要在剩余时间内完成剩余 EV 任务，则当前至少需要维持的任务底线充电功率。

#### 2.2 光伏修正项 `Delta_P_pv`

当前代码中，先计算：

```text
pv_ratio = pv_output / day_pv_peak
pv_support = clip((pv_ratio - pv_mid_ratio) / (pv_high_ratio - pv_mid_ratio), 0, 1)
Delta_P_pv = 0.60 * algo_pv_bias * max_positive_power * pv_support
```

其中：

- `pv_high_ratio = 0.50`
- `pv_mid_ratio = 0.15`
- `algo_pv_bias`
  - DDQN: `0.9`
  - PPO: `0.75`
  - 其他：`1.0`

该项表示：当前光伏相对当天峰值越高，EV 越应在任务底线之外额外增加充电功率以吸纳光伏。

#### 2.3 进度补偿项 `Delta_P_prog`

当前代码中，先定义：

```text
slack = expected_completion - completion_ratio
prog_support = clip(slack / 0.20, 0, 1)
```

若已临近日末且仍有剩余任务，则：

```text
prog_support = max(prog_support, 0.40)
```

最终：

```text
Delta_P_prog = 0.50 * max_positive_power * prog_support
```

该项表示：当 EV 当前完成进度落后于期望完成曲线时，应再向上补一部分充电功率；若接近日末，则提高兜底补电强度。

#### 2.4 削峰修正项 `Delta_P_peak`

当前代码中，若满足：

```text
current_net_load > (1 + eta_L) * day_mean_net_load
and SOC > 0.50
```

则先计算：

```text
discharge_limit = min(abs(P_ev_min), (SOC - 0.50) * E_ev / 0.25)
net_load_ratio = clip((current_net_load - day_mean_net_load) / (day_peak_net_load - day_mean_net_load), 0, 1)
peak_reduction = min(discharge_limit, k_discharge * abs(P_ev_min) * net_load_ratio)
Delta_P_peak = -peak_reduction
```

其中当前参数口径为：

- `eta_L = 0.10`
- `k_discharge = 0.50`
- 放电 SOC 阈值：`0.50`

该项是一个负向修正项，其作用不是单独定义“最终放电功率”，而是把 EV 原始目标功率向放电方向拉动。当其绝对值足够大时，最终 `target_power_raw` 会由正变负，EV 才真正进入放电削峰状态。

### 3. 当前放电逻辑说明

当前放电逻辑已不再使用旧版本固定绝对值条件：

```text
pv_output < 1.0
current_load > 0.8
SOC > 0.4
```

当前代码采用的是：

1. 基于当天净负荷序列计算：
   - `day_mean_net_load`
   - `day_peak_net_load`
2. 用当前净负荷 `current_net_load` 与当天基准净负荷比较；
3. 只有当净负荷显著高于基准，且 EV 的 `SOC > 0.50` 时，EV 才允许通过 `Delta_P_peak` 参与放电削峰。

因此，当前 EV 放电是：

```text
基于净负荷超额程度的连续放电修正机制
```

而不是旧版本的固定阈值二元触发。

### 4. 当前 fallback 逻辑说明

当前代码中，EV fallback 已改为显式开关：

```text
enable_ev_fallback = False
```

默认关闭，适合论文实验与算法公平对比。

只有当显式传入：

```text
enable_ev_fallback = True
```

时，`DDQN` 和 `PPO` 在 `action != 3` 且任务未完成的情况下，才会启用保底自动补电逻辑。该 fallback 仍只承担“保底充电”功能，不承担放电削峰功能。

### 5. 当前奖励层与决策层口径同步情况

当前 EV 奖励中的光伏判断，也已与主决策层统一为相对光伏比例判断，即基于：

```text
reward_pv_ratio = pv_output / day_pv_peak
```

进行判断，而不再使用固定原始阈值 `5.0 / 1.0 / 1.5`。

因此当前 EV 相关逻辑在代码层已经形成统一口径：

1. 充电场景判断：按 `pv_ratio`；
2. 放电削峰判断：按相对净负荷超额程度；
3. 奖励层光伏判断：按 `reward_pv_ratio`；
4. fallback：默认关闭，仅作可选保底机制。
