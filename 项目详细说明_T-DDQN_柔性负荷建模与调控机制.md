# 基于 T-DDQN 的多柔性负荷协同调控项目详细说明

## 摘要

本文档面向当前项目的代码实现，对台区多柔性负荷协同调控系统进行系统化、论文式的技术说明。全文重点围绕以下几个方面展开：第一，说明台区综合能源系统的组成及其优化目标；第二，详细建立空调负荷、电动汽车负荷、可平移负荷以及储能单元的物理模型；第三，将整个调控问题形式化为马尔可夫决策过程，并详细解释状态空间、动作空间和奖励函数的设计逻辑；第四，重点说明当前项目中 T-DDQN 算法的结构、训练方式与实际控制流程；第五，结合环境实现解释系统在每个调度步长内是如何完成从状态感知、动作决策到物理执行与奖励反馈的闭环计算。

从当前仓库实现看，本项目的核心环境位于 `rural_env.py`，T-DDQN 算法主体位于 `DRL.py`，训练器位于 `ddqn_trainer.py`，当前一键式主运行入口更接近 `train_simplified.py`。系统本质上是一个融合了物理约束、时序特征建模、多目标优化和工程启发式规则的智能调控框架。

## 1. 研究背景与项目目标

### 1.1 研究背景

随着分布式光伏在低压台区中的渗透率不断提高，传统配电台区的运行方式正从“单向供电、被动供需匹配”逐步演化为“多源接入、双向功率流、主动协同调度”的新型运行形态。在这一过程中，台区面临以下几类典型问题：

1. 光伏出力存在显著波动性和随机性，易造成中午时段局部过剩与弃光。
2. 台区基础负荷具有明显的时段性，高峰时段易产生较高购电成本与峰值负荷压力。
3. 用户侧柔性负荷逐渐增多，如空调、电动汽车和可时移家电，为主动调控提供了新的调节资源。
4. 储能的引入使台区具备更强的时移能力，但也带来了 SOC 约束、充放电效率和寿命约束等问题。
5. 单一控制目标已不能满足实际需求，必须同时权衡经济性、光伏消纳、用户舒适度和调峰效果。

因此，项目的核心任务不是单独优化某一种负荷，而是在光伏、基础负荷、柔性资源和储能共同作用的条件下，构建一个面向台区综合运行性能的智能调度策略。

### 1.2 项目目标

本项目希望通过强化学习实现以下多目标协同优化：

1. 降低台区从主网购电的综合成本。
2. 提高光伏本地消纳率，减少弃光。
3. 实现削峰填谷，抑制台区总负荷的波动性。
4. 保证用户舒适性和负荷服务质量，特别是空调舒适度和 EV 充电任务完成度。
5. 在储能约束与柔性负荷约束下获得稳定、可执行的控制策略。

上述目标并不是相互独立的。实际调度中常常存在显著冲突。例如，强行削峰可能降低空调舒适度；过度追求充电任务完成可能提升高峰购电成本；过度吸纳光伏可能导致储能 SOC 长期偏高。因此，本项目本质上是一个多目标、强约束、动态耦合的序列决策问题。

## 3. 台区调控问题的数学描述

设离散调度时刻集合为：

公式（1）：`T = {1, 2, ..., 96}`

其中一个调度周期为一天，共 96 个时段，每个时段长度为 15 分钟，即：

公式（2）：`delta_t = 0.25 h`

在任一时刻 \(t \in \mathcal{T}\)，台区总负荷可表示为：

公式（3）：`P_total(t) = P_base(t) + P_ac(t) + P_ev(t) + P_shift(t) + P_bat(t)`

其中：

- \(P_t^{base}\) 为基础负荷；
- \(P_t^{ac}\) 为空调柔性调节功率；
- \(P_t^{ev}\) 为 EV 充放电功率；
- \(P_t^{shift}\) 为可平移负荷调整量；
- \(P_t^{bat}\) 为储能充放电功率。

同时，净购电功率可写为：

公式（4）：`P_grid(t) = P_total(t) - P_pv_use(t)`

其中 \(P_t^{pv,use}\) 表示被直接消纳或经储能吸纳后有效利用的光伏功率。

系统优化目标并非单一最小化 \(P_t^{grid}\)，而是对成本、弃光、舒适度、平滑性和任务完成情况进行综合优化。

## 4. 柔性负荷物理模型构建

## 4.1 通用柔性负荷建模框架

项目首先定义了一个通用柔性负荷类 `FlexibleLoad`。该类并不对应某个具体设备，而是作为空调和 EV 这类连续调节资源的父类。其核心变量包括：

| 符号 | 代码变量 | 含义 |
|---|---|---|
| \(P^{max}\) | `max_power` | 最大允许功率 |
| \(P^{min}\) | `min_power` | 最小允许功率 |
| \(T^{resp}\) | `response_time` | 响应时间 |
| \(R^{ramp}\) | `ramp_rate` | 爬坡速率 |
| \(P_t\) | `current_power` | 当前功率 |
| \(P_t^{tar}\) | `target_power` | 目标功率 |

控制思想不是让设备瞬时跳到目标值，而是通过受爬坡率限制的渐进更新方式逼近目标功率。其离散更新公式可写为：

公式（5）：`delta_P(t) = clip(P_target(t) - P(t), -delta_P_max(t), delta_P_max(t))`

其中：

公式（6）：`delta_P_max(t) = R_ramp * delta_t / T_resp`

于是下一时刻功率为：

公式（7）：`P(t+1) = P(t) + delta_P(t)`

该建模方式具有以下优点：

1. 保证调控行为具备物理可执行性。
2. 避免负荷功率发生不合理的瞬时突变。
3. 为舒适度与时序稳定性提供了自然约束。

## 4.2 空调柔性负荷模型

### 4.2.1 建模对象与调控意义

空调负荷是典型的热惰性负荷，其柔性来自于室内热舒适具有一定容忍区间，因而空调功率可以在舒适度约束内进行时序调节。对于台区调控而言，空调负荷可以承担以下角色：

1. 在峰时适度减载，参与削峰。
2. 在低价时段或光伏富余时段适度增载，参与填谷或消纳光伏。
3. 通过室温动态实现“用热惯性换电能时移”的效果。

### 4.2.2 参数设定

当前项目中空调柔性负荷的参数如下：

- 最大功率：\(P_{ac}^{max}=15.0\)
- 最小功率：\(P_{ac}^{min}=-10.0\)
- 响应时间：\(T_{ac}^{resp}=3\)
- 爬坡速率：\(R_{ac}^{ramp}=2.0\)
- 舒适温度：\(T_{ac}^{set}=26.0^\circ C\)
- 初始室温：\(T_{ac,0}=25.0^\circ C\)

### 4.2.3 室温动态方程

项目中采用一阶简化热模型描述空调作用下的室温变化：

公式（8）：`dT_in(t)/dt = -0.1 * [P_ac(t) / P_ac_max] + 0.05 * [T_out(t) - T_in(t)]`

离散化后得到：

公式（9）：`T_in(t+1) = T_in(t) + { -0.1 * [P_ac(t) / P_ac_max] + 0.05 * [T_out(t) - T_in(t)] } * delta_t`

其中：

- \(T_t^{in}\) 为室内温度；
- \(T_t^{out}\) 为室外温度；
- 第一项反映空调制冷功率的降温效应；
- 第二项反映环境热交换。

此外，项目将室内温度约束在：

公式（10）：`16 <= T_in(t) <= 30`

### 4.2.4 舒适度建模

用户热舒适度根据室内温度与舒适设定温度的偏差进行计算：

公式（11）：`delta_T(t) = abs( T_in(t) - T_set )`

公式（12）：`C_thermal(t) = exp( -0.1 * delta_T(t) )`

该函数满足以下特征：

1. 当室温接近舒适温度时，舒适度接近 1。
2. 当室温偏离增大时，舒适度按指数衰减。
3. 该设计比线性惩罚更能体现用户对舒适边界的敏感性。

### 4.2.5 模型解释

空调模型的本质是一个“功率调节 - 室温动态 - 舒适度反馈”的闭环系统。调低空调负荷可以降低台区负荷，但会逐渐推高室温并降低舒适度；调高空调负荷可以提升舒适度或吸纳低价电/光伏，但也会抬高当前负荷。因此，空调柔性调控本质上体现了热舒适约束下的负荷时移。

### 4.2.6 空调在综合满意度中的作用

在当前代码实现中，空调并不是单独定义一个“空调满意度奖励”后直接加入总奖励，而是先将热舒适度与 EV 可用性共同融合为综合用户满意度，再参与总奖励计算。其核心计算过程如下。

首先定义当前温度偏差：

公式（12-1）：`temp_diff(t) = abs( T_in(t) - T_set )`

随后根据指数函数计算热舒适度：

公式（12-2）：`C_thermal(t) = exp( -0.1 * temp_diff(t) )`

然后根据不同时间段动态调整热舒适度权重。

峰时：

公式（12-3）：`w_th_peak = thermal_comfort_weight * peak_hour_weight`

低谷时：

公式（12-4）：`w_th_valley = thermal_comfort_weight * valley_hour_weight`

平时：

公式（12-5）：`w_th_normal = thermal_comfort_weight`

在当前实现中，默认参数大致为：

- `thermal_comfort_weight = 0.5`
- `discharge_willingness_weight = 0.5`
- `peak_hour_weight = 0.3`
- `valley_hour_weight = 0.7`

因此：

1. 在峰时，热舒适度权重会下降，意味着系统更倾向于优先保障供电与削峰。
2. 在低谷时，热舒适度权重会上升，意味着系统更愿意兼顾用户体验。
3. 在平时，采用基础权重。

综合满意度的计算形式可以写为：

公式（12-6）：`S_user(t) = w_th(t) * C_thermal(t) + w_ev(t) * SOC_ev(t)`

其中：

公式（12-7）：`w_ev(t) = 1 - w_th(t)`，在峰时和谷时成立；

在平时段，当前代码直接取：

公式（12-8）：`w_ev_normal = discharge_willingness_weight`

这意味着空调热舒适度并不是孤立评估的，而是与 EV 的可用性一起共同构成“用户侧满意度”。从物理意义上说，这样的设计体现了一个用户侧综合偏好：一方面用户希望室内温度舒适，另一方面也希望车辆电量可用，系统需要在二者之间动态分配权重。

## 4.3 电动汽车柔性负荷模型

### 4.3.1 建模对象与调控意义

电动汽车在本项目中被视为兼具“可调充电负荷”和“分布式储能单元”双重属性的柔性资源。其控制目标不只是改变某一时刻的充电功率，更重要的是在满足用户充电需求的前提下，利用 EV 的可延迟性和可反向放电能力协助台区运行优化。

### 4.3.2 参数设定

当前 EV 模型参数如下：

- 最大充电功率：\(P_{ev}^{max}=20.0\)
- 最大放电功率下界：\(P_{ev}^{min}=-15.0\)
- 响应时间：\(T_{ev}^{resp}=8\)
- 爬坡速率：\(R_{ev}^{ramp}=1.5\)
- 初始荷电状态：\(SOC_{ev,0}=0.6\)
- 电池容量：\(E_{ev}=200.0\)
- 放电允许阈值：\(SOC_{ev}^{dis}=0.5\)

### 4.3.3 SOC 动态模型

当 EV 充电时：

公式（13）：`SOC_ev(t+1) = SOC_ev(t) + P_ev(t) * delta_t / E_ev`，条件：`P_ev(t) > 0`

当 EV 放电时：

公式（14）：`SOC_ev(t+1) = SOC_ev(t) - abs(P_ev(t)) * delta_t / E_ev`，条件：`P_ev(t) < 0`

并施加边界：

公式（15）：`0 <= SOC_ev(t) <= 1`

### 4.3.4 放电约束

项目规定仅当：

公式（16）：`SOC_ev(t) >= 0.5`

时 EV 才允许参与放电。这一约束体现了用户侧可接受性原则，即 EV 不能为了台区调控而过度牺牲车辆续航能力。

### 4.3.5 EV 满意度与任务约束

EV 调控不仅受 SOC 上下界约束，还隐含一个“最终充电任务完成”的需求。代码中专门设计了 EV 能量完成奖励和进度奖励，用于鼓励在日末或任务期末达到合理荷电状态。

这意味着 EV 模型不是单纯的电池模型，而是“可调储能 + 任务完成”的组合模型。它既能在峰时放电参与削峰，也必须在合适时段及时回补电量，以保证用户可用性。

### 4.3.5.1 EV 参考任务能量的定义

当前实现中，EV 的“任务层”不是直接规定某辆车在某个时刻必须达到某个 SOC，而是先从真实 EV 参考负荷曲线中计算“当天总充电任务量”。若某一天参考 EV 充电功率曲线为 `real_ev_load_day(k)`，则参考总能量定义为：

公式（16-1）：`E_ev_ref = sum( max(real_ev_load_day(k), 0) ) * 0.25`

其中：

1. `max(real_ev_load_day(k), 0)` 表示只统计正向充电功率；
2. `0.25` 表示每个时段长度为 0.25 小时；
3. `E_ev_ref` 的单位为 kWh。

这意味着项目把“真实参考 EV 负荷”抽象成了“当天应完成的总充电能量任务”。

### 4.3.5.2 已完成能量与剩余能量

环境中还维护两个任务层状态量：

1. 实际已完成充电能量 `E_ev_done(t)`  
2. 剩余待完成充电能量 `E_ev_remain(t)`

其更新关系可以表示为：

公式（16-2）：`E_ev_done(t+1) = min( E_ev_ref, E_ev_done(t) + max(P_ev(t), 0) * delta_t )`

公式（16-3）：`E_ev_remain(t) = max( 0, E_ev_ref - E_ev_done(t) )`

这两个量的物理意义非常明确：

1. `E_ev_done(t)` 表示截至当前时刻，系统已经为 EV 累计补充了多少能量；
2. `E_ev_remain(t)` 表示为了完成当天 EV 任务，还剩多少能量必须在后续时段完成。

因此，EV 在当前环境中的任务层不是“任意充放电”，而是始终被一个总能量守恒目标牵引。

### 4.3.5.3 期望充电进度曲线

为了防止策略把所有 EV 充电任务都拖到一天最后，环境定义了一个“期望完成进度曲线”。设 `t_step` 为当天第几个 15 分钟步长，则期望完成比例 `r_exp(t_step)` 分段定义如下：

公式（16-4）：当 `t_step < 24` 时，`r_exp(t_step) = 0.0`

公式（16-5）：当 `24 <= t_step < 48` 时，`r_exp(t_step) = 0.3 * (t_step - 24) / 24`

公式（16-6）：当 `48 <= t_step < 72` 时，`r_exp(t_step) = 0.3 + 0.45 * (t_step - 48) / 24`

公式（16-7）：当 `72 <= t_step < 84` 时，`r_exp(t_step) = 0.75 + 0.15 * (t_step - 72) / 12`

公式（16-8）：当 `84 <= t_step < 96` 时，`r_exp(t_step) = min(1.0, 0.9 + 0.1 * (t_step - 84) / 12 )`

其对应的物理含义是：

1. 凌晨 0:00-6:00 不要求完成充电进度；
2. 6:00-12:00 逐步完成到 30%；
3. 12:00-18:00 逐步完成到 75%；
4. 18:00-21:00 完成到 90%；
5. 最后 3 小时补完剩余 10%。

这条曲线体现了当前项目明显的“偏向白天充电、偏向光伏窗口充电”的策略先验。

### 4.3.5.4 当前完成度与进度偏差

根据参考总能量与已完成能量，可以定义当前实际完成比例：

公式（16-9）：`r_cur(t) = E_ev_done(t) / E_ev_ref`

然后定义进度偏差：

公式（16-10）：`gap_ev(t) = r_exp(t) - r_cur(t)`

当 `gap_ev(t) > 0` 时，表示当前充电进度落后于期望；  
当 `gap_ev(t) < 0` 时，表示当前进度领先于期望。

### 4.3.5.5 EV 进度奖励

环境通过一个专门的“进度奖励”对齐期望进度曲线。其形式为：

若进度落后：

公式（16-11）：`R_ev_progress(t) = -2.5 * gap_ev(t)`，条件是 `gap_ev(t) > 0`

若进度超前：

公式（16-12）：`R_ev_progress(t) = min(1.0, -gap_ev(t) * 1.5)`，条件是 `gap_ev(t) <= 0`

这意味着：

1. 落后期望进度会受到惩罚；
2. 略微超前会获得奖励；
3. 奖励被限制在 1.0 以内，避免“过早超额充电”获得过高回报。

### 4.3.5.6 EV 任务完成奖励

除了过程中的进度奖励外，环境还设计了一个终局性质更强的“总任务完成奖励”。定义完成度比例：

公式（16-13）：`r_full(t) = E_ev_done(t) / E_ev_ref`

则 EV 任务完成奖励按如下分段计算。

如果完成度在合理区间 `[0.9, 1.1]` 内：

公式（16-14）：`R_ev_full(t) = 2.0 + [ 1.0 - abs(r_full(t) - 1.0) / 0.1 ]`

如果充电不足，即 `r_full(t) < 0.9`：

公式（16-15）：`R_ev_full(t) = -5.0 * (0.9 - r_full(t))`

若严重不足，即 `r_full(t) < 0.7`，还会追加惩罚：

公式（16-16）：`R_ev_full(t) = -5.0 * (0.9 - r_full(t)) - 5.0 * (0.7 - r_full(t))`

如果过度充电，即 `r_full(t) > 1.1`：

公式（16-17）：`R_ev_full(t) = -2.0 * (r_full(t) - 1.1)`

该设计反映出：

1. EV 最优不是“越多充越好”，而是“尽量接近参考任务量”；
2. 充电不足惩罚更重，因为它直接影响用户可用性；
3. 过度充电也不是理想行为，因为这可能造成不必要的负荷增加、时间浪费和电池寿命损耗。

### 4.3.5.7 EV 在综合满意度中的体现

当前代码并没有单独构造一个“EV 满意度函数”并直接对其指数化，而是把 EV 的可用性通过 `SOC_ev(t)` 直接并入综合用户满意度。其形式为：

公式（16-18）：`S_user(t) = w_th(t) * C_thermal(t) + w_ev(t) * SOC_ev(t)`

这意味着 EV 对“用户满意度”的贡献主要通过以下方式体现：

1. EV 的 SOC 越高，用户越有安全感和可用性保障；
2. EV 的 SOC 越低，用户满意度越差；
3. 在峰时和谷时，EV 在综合满意度中的权重会随热舒适权重联动调整。

因此，EV 的“满意度层”在当前项目里不是孤立定义的，而是与空调热舒适共同组成统一的用户侧综合满意度指标。

### 4.3.6 模型解释

EV 模型体现了台区调控中的一个核心矛盾：  
若过度利用 EV 放电，虽可降低峰时负荷，却可能导致最终充电不足；若过度强调充电任务完成，又会削弱其对台区削峰填谷和光伏消纳的支撑作用。因此，EV 调控是一个典型的用户目标与系统目标耦合问题。

### 4.3.7 EV 充放电目标功率的论文式写法与代码对应

前述 4.3.5 主要说明了 EV 的任务层、进度层和满意度层，但若从“动作 3 如何具体形成充放电功率”这一角度看，还需要补充更接近论文表达的目标功率分解。

从论文写法角度，可以将 EV 目标功率抽象写为：

公式（16-19）：`P_ev_target(t) = P_ev_base(t) + Delta_P_ev_pv(t) + Delta_P_ev_load(t) + Delta_P_ev_comp(t)`

其中：

1. `P_ev_base(t)` 表示参考充电基准项；
2. `Delta_P_ev_pv(t)` 表示光伏感知补偿项；
3. `Delta_P_ev_load(t)` 表示负荷感知补偿项；
4. `Delta_P_ev_comp(t)` 表示进度偏差补偿项。

需要特别说明的是：

**当前仓库代码并不是严格以这一条闭式公式直接计算 EV 目标功率，而是采用“分支逻辑 + 限幅 + 平滑”的工程实现方式。**

但是，如果从机理层面总结，代码中的 EV 充放电逻辑与上述四项分解是高度对应的。

#### 4.3.7.1 基准充电项

若以参考 EV 历史负荷为基准，则论文中可写：

公式（16-20）：`P_ev_base(t) = EV_His_Data(t, d)`

其中 `EV_His_Data(t, d)` 表示日期 `d` 在时刻 `t` 的参考 EV 充电功率。

在当前代码中，没有直接把整条历史参考功率曲线作为动作 3 的实时基准项，而是将其进一步积分成“总任务能量” `E_ev_ref`，再通过 `required_power` 把剩余任务均匀分摊到后续时段。因此，更贴近当前代码的基准项可写为：

公式（16-21）：`P_ev_base(t) = P_ev_req(t) = min( P_ev_max_pos(t), E_ev_remain(t) / (N_remain(t) * delta_t) )`

其中：

1. `P_ev_max_pos(t)` 为当前允许的最大正向充电功率；
2. `E_ev_remain(t)` 为当前剩余待充能量；
3. `N_remain(t)` 为从当前时刻到日末的剩余时间步数。

这表示：如果后面所有时间步都平均分担任务，那么当前至少应当承担 `P_ev_req(t)` 这一基准充电功率。

#### 4.3.7.2 光伏感知充电项

论文式写法通常会显式写成：

公式（16-22）：`Delta_P_ev_pv(t) = k_charge * P_ev_max * f_pv(t)`

其中 `f_pv(t)` 是一个由光伏出力强度决定的归一化函数。

若进一步贴近当前代码逻辑，可将其写为：

公式（16-23）：`P_ev_pv_priority(t) = min( P_ev_max_pos(t), beta_pv * P_pv(t) )`

这里当前代码中：

1. 在动作 3 分支里，常取 `beta_pv = 0.85`；
2. 在自动补电逻辑里，常取 `beta_pv = 0.9`；
3. 不同算法还会再乘一个 `algo_pv_bias`，例如 DDQN 和 PPO 会适度降低光伏驱动敏感性。

因此，代码中的光伏感知项本质上是：

公式（16-24）：`Delta_P_ev_pv(t) ≈ algo_pv_bias * min( P_ev_max_pos(t), beta_pv * P_pv(t) )`

它表达的物理意义是：当光伏出力较高时，EV 应优先提高充电功率，以吸纳更多本地光伏。

#### 4.3.7.3 负荷感知放电项

前面的 4.3.3 和 4.3.4 说明了 EV 可以放电以及放电的 SOC 边界，但还没有把“什么时候主动放电削峰”写成论文式目标功率项。对此可抽象为：

公式（16-25）：`Delta_P_ev_load(t) = - k_discharge * P_ev_dis_max * f_load(t)`

其中 `f_load(t)` 表示当前系统负荷紧张程度函数，值越大说明越应参与放电削峰。

更贴近当前代码逻辑，可以写为：

公式（16-26）：当 `L_net(t) > (1 + eta_L) * L_base_net(t)` 且 `SOC_ev(t) > 0.5` 时，

`P_ev_dis_limit(t) = min( P_ev_dis_max, (SOC_ev(t) - 0.5) * E_ev / delta_t )`

公式（16-27）：`P_ev_peak_reduce(t) = min( P_ev_dis_limit(t), k_discharge * P_ev_dis_max * min( (L_net(t) - L_base_net(t)) / (L_peak_net(t) - L_base_net(t)), 1.0 ) )`

公式（16-28）：`Delta_P_ev_load(t) = - P_ev_peak_reduce(t)`

这里的物理逻辑是：

1. 当光伏很低时，说明系统难以依靠光伏支撑负荷；
2. 当基础负荷较高时，系统存在削峰需求；
3. 当 EV SOC 足够高时，EV 具备可放电余量；
4. 此时允许 EV 通过负向功率参与削峰。

因此，当前代码中的 EV 放电不是全天候放电，而是一种在“低光伏 + 高负荷 + SOC 充足”条件下触发的削峰机制。

#### 4.3.7.4 进度补偿项

进度补偿项对应你上传图片中的核心思想，也是当前项目 EV 任务层最关键的一部分。论文式可写为：

公式（16-29）：`Delta_P_ev_comp(t) = k_comp * P_ev_max * [ r_exp(t) - r_cur(t) ]`

其中：

1. `r_exp(t)` 为目标进度；
2. `r_cur(t)` 为当前已完成进度；
3. 二者差值表示当前进度偏差。

不过当前代码并没有直接把 `Delta_P_ev_comp(t)` 按这一条线性表达式显式加到目标功率上，而是采用“根据进度偏差调节 required_power 是否必须被满足、以及在低光伏时是否允许兜底充电”的逻辑实现。

更贴近当前代码的理解是：

1. 当 `gap_ev(t) = r_exp(t) - r_cur(t)` 较大时，系统会提高对 `required_power` 的满足优先级；
2. 当 `gap_ev(t)` 严重落后，且已接近日末时，即便光伏较低，也允许触发补电兜底；
3. 当进度并不落后时，系统会减少夜间或低光伏时段的无效充电。

因此，当前代码里的 `Delta_P_ev_comp(t)` 更像一个**通过条件分支隐式实现的补偿项**。

#### 4.3.7.5 综合理解与论文写法建议

因此，如果要用论文语言总结当前 EV 动作 3 的机理，可以表述为：

1. EV 目标功率由基准任务项、光伏感知项、负荷削峰项和进度补偿项共同决定；
2. 在光伏充足时，EV 优先承担光伏吸纳任务；
3. 在低光伏高负荷且 SOC 充足时，EV 允许参与放电削峰；
4. 在进度落后时，系统通过补偿机制提高 EV 充电优先级；
5. 最终目标功率还需经过功率边界、SOC 边界和平滑控制约束。

这也说明，你上传图片中的那套论文写法，与当前项目 EV 充放电设计**在思想上是对应的**，但与当前代码实现**不是逐项一模一样的一一闭式映射**。更准确地说：

**图片中的写法是“论文归纳表达”，当前代码是“工程化分支实现”。**

前者便于写论文和阐释机理，后者便于在复杂场景下稳定运行。

## 4.4 可平移负荷模型

### 4.4.1 建模思想

可平移负荷并不强调设备级动态，而强调聚合层面的时序重排能力。其核心特点是：

1. 总能量需求不变。
2. 某些用电任务可以提前或延后执行。
3. 当前时段负荷的增减会在未来时段以某种方式补偿回来。

### 4.4.2 参数设定

当前环境中定义：

- 当前平移功率：\(P_{shift,t}\)
- 最大平移功率：\(P_{shift}^{max}=25.0\)
- 平移延迟变量：`shift_delay`

### 4.4.3 调控方式

当系统判断当前适合填谷或吸纳光伏时，可令：

公式（17）：`P_shift(t) > 0`

表示将未来可转移任务前移到当前执行。

当系统判断当前适合削峰时，可令：

公式（18）：`P_shift(t) < 0`

表示将当前一部分可延迟任务后移。

同时有边界约束：

公式（19）：`-P_shift_max <= P_shift(t) <= P_shift_max`

此外，环境还对相邻时刻平移负荷变化施加平滑约束，避免出现不合理的负荷跳变。

### 4.4.4 模型解释

与空调或 EV 不同，可平移负荷不通过内部物理储能体现柔性，而是通过“时间重排”体现柔性。它可以被理解为一种虚拟储能：并不储存电能，而是储存“任务执行时机”。

## 4.5 储能系统模型

### 4.5.1 建模作用

储能系统在项目中承担三类关键功能：

1. 高峰时放电，降低购电和总负荷峰值。
2. 低谷时或低价时充电，实现经济套利。
3. 光伏富余时吸收弃光，提升本地消纳率。

### 4.5.2 参数设定

环境中储能参数如下：

- 电池容量：\(E_b=100.0\text{ kWh}\)
- 最大充电功率：\(P_b^{ch,max}=50.0\text{ kW}\)
- 最大放电功率：\(P_b^{dis,max}=50.0\text{ kW}\)
- 初始荷电状态：\(SOC_{b,0}=0.5\)
- 充放电效率：\(\eta=0.9\)
- 安全运行范围：\(0.15 \le SOC_{b,t} \le 0.85\)

### 4.5.3 SOC 动态方程

储能充电时：

公式（20）：`SOC_b(t+1) = SOC_b(t) + P_b_ch(t) * eta * delta_t / E_b`

储能放电时：

公式（21）：`SOC_b(t+1) = SOC_b(t) - P_b_dis(t) * delta_t / (eta * E_b)`

并满足：

公式（22）：`0.15 <= SOC_b(t) <= 0.85`

### 4.5.4 光伏优先吸纳机制

若当前时刻光伏出力大于负荷需求，则会产生弃光候选量：

公式（23）：`P_curt(t) = max(0, P_pv(t) - P_load(t))`

储能可优先吸收其中一部分：

公式（24）：`P_bat_pv(t) = min( E_b * (0.85 - SOC_b(t)) / eta, P_curt(t), P_b_ch_max )`

该机制使储能成为连接光伏波动与负荷需求的重要缓冲器。

### 4.5.5 模型解释

储能模型的关键不是简单描述电池，而是使台区调控具备“能量跨时段搬运”的能力。空调与可平移负荷属于需求侧柔性，EV 属于任务约束下的双向资源，而储能则是最纯粹的时移型资源。四者协同后，整个系统的调度自由度显著提升。

### 4.5.6 储能的动作约束与运行边界

在当前环境中，储能并不是只通过 SOC 方程被动更新，而是还受到动作层的显式约束。对于放电动作，环境实际执行的放电功率满足：

公式（24-1）：`P_b_dis(t) <= min( P_b_dis_max, (SOC_b(t) - 0.15) * eta * E_b / delta_t, P_need(t) )`

对于充电动作，环境满足：

公式（24-2）：`P_b_ch(t) <= min( P_b_ch_max, (0.85 - SOC_b(t)) * E_b / (eta * delta_t), P_avail(t) )`

其中：

1. 第一项是功率额定上限；
2. 第二项来自 SOC 边界；
3. 第三项来自系统当前场景需要，例如可用光伏剩余或低价窗口。

这说明储能调控不是任意充放，而是被“设备能力 + 安全边界 + 场景需求”三重机制共同约束。

### 4.5.7 储能在奖励中的作用

储能不会直接单独生成一个“电池奖励函数”，而是通过多个路径影响总奖励：

1. 通过降低高价购电影响经济奖励；
2. 通过吸纳弃光影响光伏消纳奖励；
3. 通过调平净负荷影响削峰填谷奖励；
4. 通过维持合理 SOC 获得 SOC 平衡奖励或避免 SOC 惩罚。

因此，储能在本项目中是一个典型的多目标耦合资源，而不是只做电价套利。

## 4.6 分布式光伏的处理机制

### 4.6.1 光伏在系统中的角色

在当前项目中，分布式光伏被视为外部不可控输入，其出力由数据文件直接给定。强化学习智能体并不直接控制光伏发电大小，而是通过调节储能、空调、EV 和可平移负荷来改变光伏的消纳效果。

因此，光伏在系统中的关键不是“怎么发”，而是“怎么用好、怎么少弃光、怎么与柔性负荷耦合”。

### 4.6.2 光伏直接消纳与弃光

当前时刻，负荷对光伏的直接消纳量定义为：

公式（24-3）：`P_pv_used_direct(t) = min( P_pv(t), P_load(t) )`

若光伏出力高于当前负荷需求，则产生初始弃光候选量：

公式（24-4）：`P_curt_init(t) = max( 0, P_pv(t) - P_pv_used_direct(t) )`

这一步表达的物理意义是：

1. 负荷能先直接吃掉一部分光伏；
2. 多出来的部分若没有其他资源吸收，就会形成弃光风险。

### 4.6.3 储能吸纳弃光

环境随后会调用储能吸纳一部分弃光。当前实现中，一个近似表达可以写为：

公式（24-5）：`P_bat_from_pv(t) = min( P_curt_init(t) * 0.8, P_b_ch_max, SOC_room(t) )`

其中：

公式（24-6）：`SOC_room(t) = E_b * (0.85 - SOC_b(t)) / eta`

随后最终弃光量为：

公式（24-7）：`P_curt_final(t) = P_curt_init(t) - P_bat_from_pv(t)`

这意味着当前项目并不是简单把“光伏减去负荷”当作最终利用结果，而是允许储能继续吸收原本会被弃掉的光伏。

### 4.6.4 光伏利用率的定义

当前代码对光伏利用率的计算比“直接利用量 / 总出力”更细。它把“直接被负荷消耗的光伏”与“被电池吸收的弃光”都算进总利用量。

定义直接利用量：

公式（24-8）：`P_pv_use_direct(t) = min( P_pv(t), P_total(t) )`

定义储能吸纳量：

公式（24-9）：`P_pv_use_store(t) = P_bat_from_pv(t)`

则总光伏利用量为：

公式（24-10）：`P_pv_use_total(t) = min( P_pv_use_direct(t) + P_pv_use_store(t), P_pv(t) )`

于是基础利用率为：

公式（24-11）：`rho_pv_base(t) = P_pv_use_total(t) / max( P_pv(t), 0.05 )`

### 4.6.5 低出力平滑与 EWMA 处理

为了避免光伏出力极小时利用率发生数值抖动，环境并没有直接使用 `rho_pv_base(t)` 作为最终利用率，而是引入了两层平滑：

第一层是低出力平滑因子：

公式（24-12）：`rho_pv_smooth(t) = rho_pv_base(t) * smooth_factor(t)`，当 `P_pv(t) < 8.0` 时生效

第二层是指数加权移动平均：

公式（24-13）：`rho_pv_ewma(t) = alpha * rho_pv_smooth(t) + (1 - alpha) * rho_pv_ewma(t-1)`

当前代码中 `alpha ≈ 0.3`。  
最终输出利用率为：

公式（24-14）：`rho_pv(t) = clip( rho_pv_ewma(t), 0, 1 )`

这说明项目中的光伏利用率不是一个原始瞬时比值，而是一个经过稳定化处理的调度指标，更适合进入状态空间和奖励函数。

### 4.6.6 光伏在奖励中的作用

光伏相关收益主要通过两条路径进入奖励：

第一条是消纳奖励与弃光惩罚：

公式（24-15）：`R_pv(t) = bonus_pv(t) * P_pv_used_direct(t) + penalty_curt * P_curt_final(t)`

其中 `penalty_curt < 0`，表示弃光越多惩罚越大。

第二条是光伏-负荷匹配奖励，即鼓励总负荷在光伏高发时段更贴近光伏出力：

公式（24-16）：`R_match(t) = exp( -0.1 * abs( P_total(t) - P_pv(t) ) )`

该项意味着：

1. 光伏高时，若总负荷也适度提高，则匹配度更好；
2. 光伏低时，若仍维持过高负荷，则匹配度下降；
3. 因此系统会倾向于把可调资源向光伏窗口靠拢。

### 4.6.7 光伏与各类柔性资源的耦合方式

从控制逻辑上看，光伏并不是孤立参与优化，而是通过以下方式影响各资源动作：

1. 影响储能是否优先充电；
2. 影响空调是否在中午适度增载；
3. 影响 EV 是否在白天优先补能；
4. 影响可平移负荷是否前移到光伏高发时段。

因此，项目中所谓“分布式光伏处理”，本质上并不是对光伏本身做控制，而是围绕光伏构建了一个“负荷跟着光伏走、储能帮助吸纳弃光”的协同消纳机制。

## 5.3 状态空间设计逻辑

该状态设计不是随意拼接，而是围绕“系统是否需要调、还能怎么调、调了以后会怎样”三个核心问题展开：

1. 是否需要调  
   由负荷、电价、光伏和时刻信息决定。

2. 还能怎么调  
   由储能 SOC、EV SOC、空调功率、可平移负荷状态决定。

3. 调了以后会怎样  
   由空调室温、光伏利用率、日内进度等信息体现。

从强化学习角度看，这样的状态变量设计有助于提升可观测性，使智能体不只是知道当前负荷高低，还知道系统剩余调节自由度及其潜在后果。

## 6. 动作空间设计与控制含义

## 6.1 动作集合

项目采用离散动作空间：

公式（26）：`A = {0, 1, 2, 3, 4}`

各动作含义如下：

| 动作编号 | 含义 | 调控对象 |
|---|---|---|
| 0 | 储能放电 | Battery |
| 1 | 储能充电 | Battery |
| 2 | 调节空调设定功率 | Air Conditioning |
| 3 | 调节 EV 充放电功率 | EV |
| 4 | 调节可平移负荷 | Shiftable Load |

## 6.2 为什么采用离散动作而不是连续动作

该项目选择离散动作的原因主要包括：

1. 便于将调控任务分解为几类可解释的控制模式。
2. 与 DDQN 系列算法天然匹配。
3. 便于将工程先验嵌入动作逻辑。
4. 在当前多目标、多约束系统中，先决定调谁，再由环境根据约束计算调多少，有利于提升控制稳定性。

因此，动作 \(a_t\) 的含义不是“直接输出精确功率”，而是“选择当前最优调控方向”，而具体功率幅值由环境内部根据规则和物理约束进一步确定。

## 7. 单步调控机制的详细说明

## 7.1 调控总流程

在任一时刻 \(t\)，系统执行以下流程：

1. 读入当前外部状态  
   获取基础负荷 \(P_t^{base}\)、光伏出力 \(P_t^{pv}\)、当前时刻、电价和日期信息。

2. 形成状态向量  
   将储能 SOC、EV SOC、空调室温、可平移负荷状态等拼接成状态 \(s_t\)。

3. T-DDQN 选择动作  
   根据状态序列或当前状态输出动作类别 \(a_t\)。

4. 环境进入动作分支  
   按照动作类型执行储能、空调、EV 或可平移负荷控制。

5. 根据约束计算实际调节量  
   综合考虑功率上下限、SOC 边界、舒适度限制、光伏条件和峰谷属性。

6. 更新物理状态  
   更新储能 SOC、EV SOC、空调温度、可平移负荷功率和总负荷。

7. 计算奖励  
   根据成本、弃光、舒适度、削峰填谷效果和 EV 任务完成度构造即时奖励。

8. 进入下一时刻  
   返回下一状态 \(s_{t+1}\) 和奖励 \(r_t\)。

上述过程可抽象为：

公式（27）：`s(t) --pi_theta--> a(t) --Env--> [ s(t+1), reward(t) ]`

---

## 7.2 动作 0：储能放电

当动作选择为储能放电时，系统会判断以下条件：

1. 当前净负荷是否足够高。
2. 储能 SOC 是否高于最低安全边界。
3. 当前放电是否具有足够的削峰或经济价值。

若满足放电条件，则放电功率一般受三重约束：

公式（28）：`P_b_dis(t) <= min( P_b_dis_max, (SOC_b(t) - 0.15) * eta * E_b / delta_t, P_need(t) )`

其中 \(P_t^{need}\) 表示当前从系统运行角度所需的放电强度。

该动作的目标主要有：

1. 降低高峰时段购电功率。
2. 抑制净负荷峰值。
3. 在高电价时实现经济收益。

---

## 7.3 动作 1：储能充电

当动作选择为储能充电时，系统优先考虑以下两类情形：

1. 存在光伏富余，优先利用弃光充电。
2. 当前处于低谷低价时段，允许从电网充电。

实际充电功率满足：

公式（29）：`P_b_ch(t) <= min( P_b_ch_max, (0.85 - SOC_b(t)) * E_b / (eta * delta_t), P_avail(t) )`

其中 \(P_t^{avail}\) 表示可用于充电的光伏剩余或允许的低价电网功率。

该动作主要用于：

1. 吸纳弃光。
2. 在低价时段储能。
3. 为后续高峰削峰预留放电能力。

---

## 7.4 动作 2：空调调节

当动作选择为空调调节时，环境会综合以下因素计算目标调节量：

1. 当前基础负荷高低。
2. 当前光伏出力大小。
3. 当前时段是否处于高峰。
4. 当前室温与舒适温度的偏差。
5. 空调当前功率及其爬坡约束。

一般而言：

1. 高负荷、低光伏、高电价时，系统倾向于降低空调负荷。
2. 低负荷、光伏富余时，系统允许空调适度增载。

设目标空调功率为 \(P_{ac,t}^{tar}\)，则实际功率并不瞬时到达目标，而是通过式（5）至式（7）渐进更新。之后再通过式（9）更新室内温度。

因此，空调动作的真实影响链条是：

公式（30）：`a(t)=2 -> P_ac_target(t) -> P_ac(t+1) -> T_in(t+1) -> C_thermal(t+1)`

这表明空调控制对奖励的影响通常存在滞后性，而非一步即刻完全显现。

---

## 7.5 动作 3：EV 调节

当动作选择为 EV 调节时，系统需要判断 EV 更适合充电还是放电。其逻辑通常包括：

1. 若高峰负荷较大且 EV SOC 较高，则更倾向于放电。
2. 若低谷或光伏富余，则更倾向于充电。
3. 若日末临近且 EV 补能进度不足，则优先保证充电任务完成。

其控制约束可写为：

公式（31）：`P_ev_min <= P_ev(t) <= P_ev_max`

公式（32）：`SOC_ev(t+1) in [0, 1]`

且若执行放电还需满足：

公式（33）：`SOC_ev(t) >= 0.5`

项目中 EV 的调控目标可概括为：

1. 利用高 SOC 时段为台区提供削峰能力。
2. 利用低谷和富余光伏时段完成补能。
3. 在系统效益和用户任务之间取得平衡。

---

## 7.6 动作 4：可平移负荷调节

当动作选择为可平移负荷调节时，系统根据当前场景决定增加还是减少当前时段的可平移负荷。

若：

1. 低谷时段；
2. 光伏富余明显；
3. 当前负荷水平较低；

则倾向于令：

公式（34）：`P_shift(t) > 0`

表示前移未来任务。

若：

1. 当前处于峰时；
2. 负荷较高且光伏不足；

则倾向于令：

公式（35）：`P_shift(t) < 0`

表示后移当前任务。

并需满足平滑约束：

公式（36）：`abs(P_shift(t+1) - P_shift(t)) <= delta_P_shift_max`

该设计避免可平移负荷在相邻时间步出现过度剧烈波动。

## 8. 奖励函数设计

## 8.1 总体形式

项目采用多目标加权奖励函数。可抽象表示为：

公式（37）：`reward(t) = w1*R_econ(t) + w2*R_pv(t) + w3*R_user(t) + w4*R_peak(t) + w5*R_soc(t) + w6*R_ev(t) + R_aux(t)`

其中：

- \(R_t^{econ}\)：经济性奖励；
- \(R_t^{pv}\)：光伏消纳奖励；
- \(R_t^{user}\)：用户满意度奖励；
- \(R_t^{peak}\)：削峰填谷和平滑性奖励；
- \(R_t^{soc}\)：储能 SOC 平衡奖励与惩罚；
- \(R_t^{ev}\)：EV 充电任务奖励；
- \(R_t^{aux}\)：若干辅助奖励或约束惩罚项。

## 8.2 经济奖励

若当前购电功率为 \(P_t^{grid}\)，电价为 \(\pi_t\)，则购电成本为：

公式（38）：`C_grid(t) = price(t) * P_grid(t) * delta_t`

对应经济奖励一般取为其负值或负值缩放形式：

公式（39）：`R_econ(t) = -alpha * C_grid(t)`

该项直接鼓励降低购电量与高价时段购电行为。

## 8.3 光伏消纳奖励

若存在弃光量 \(P_t^{curt}\)，则环境对弃光给予惩罚，等价于鼓励提高光伏利用率。设光伏利用率为：

公式（40）：`rho_pv(t) = P_pv_use(t) / (P_pv(t) + epsilon)`

则可通过以下形式构建奖励：

公式（41）：`R_pv(t) = beta * rho_pv(t) - gamma * P_curt(t)`

## 8.4 用户满意度奖励

用户满意度主要由两部分组成：

1. 空调舒适度；
2. EV 荷电状态可接受度。

总体满意度可表示为：

公式（42）：`S(t) = w_th(t) * C_thermal(t) + w_ev(t) * SOC_ev(t)`

其中：

公式（43）：`w_th(t) + w_ev(t) = 1`

在不同峰谷时段，项目会动态调整舒适度权重和 EV 权重，从而体现不同运行场景下用户侧偏好的相对重要性。

## 8.5 削峰填谷奖励

环境会根据当前负荷平滑性、柔性负荷调节方向和峰谷匹配情况构造额外奖励。例如：

1. 在高峰期成功减载获得正奖励。
2. 在低谷期适度增载获得正奖励。
3. 若负荷变化更平滑，则获得额外稳定性收益。

该部分本质上鼓励策略不只是降低某一时刻功率，而是优化整条日负荷曲线。

## 8.6 EV 任务奖励

当 EV 在日末达到预期能量要求时，环境给予正奖励；若进度不足或最终未完成，则给予惩罚。这样可以避免策略为了短期削峰而持续透支 EV 用户需求。

## 9. T-DDQN 算法原理与实现

## 9.1 算法定位

本项目的 T-DDQN 可理解为一种面向时序能量调度问题的增强型 DQN。它并非只看单时刻状态，而是利用一段历史状态序列学习负荷、光伏和 SOC 的动态演化规律。

从结构上看，T-DDQN 同时融合了：

1. Double DQN  
   用于缓解 Q 值过估计。

2. Dueling Network  
   用于分离状态价值与动作优势。

3. Transformer Encoder  
   用于从状态序列中提取长期时序依赖。

因此，T-DDQN 并不是简单替换 DDQN 的网络，而是将时序建模能力显式引入到了 Q 值近似器中。

---

## 9.2 输入序列构造

与普通 DQN 仅输入当前状态 \(s_t\) 不同，T-DDQN 输入的是长度为 \(L\) 的状态序列：

公式（44）：`S_seq(t) = [ s(t-L+1), s(t-L+2), ..., s(t) ]`

当前实现中：

公式（45）：`L = 24`

由于每个时间步为 15 分钟，因此该窗口约覆盖最近 6 小时的信息。该设计的主要意义在于：

1. 捕捉日内负荷变化趋势。
2. 捕捉光伏变化趋势。
3. 捕捉储能和 EV 状态演化路径。
4. 提升对“未来一段时间”调控价值的判断能力。

此外，网络还会拼接若干储能参数，如容量、最大充电功率和最大放电功率，以增强模型对系统物理能力边界的感知。

---

## 9.3 网络结构

### 9.3.1 输入嵌入层

设序列中每个状态向量维度为 \(d_s\)，则在进入 Transformer 之前，先通过线性映射将其嵌入到高维特征空间：

公式（46）：`z(t) = W_e * s(t) + b_e`

### 9.3.2 位置编码

由于 Transformer 本身不具有时序顺序信息，需要通过位置编码加入时序位置信息：

公式（47）：`z_hat(t) = z(t) + PE(t)`

其中 \(PE_t\) 为位置编码向量。

### 9.3.3 Transformer 编码器

状态序列经过多层多头自注意力编码器后，学习时间依赖关系。设编码后序列特征为：

公式（48）：`H(t) = [ h(t-L+1), h(t-L+2), ..., h(t) ]`

该结构能够学习如下信息：

1. 当前高负荷是否来自持续爬升趋势；
2. 当前高光伏是否具有持续性；
3. 当前 SOC 水平是否会在未来若干时段形成约束；
4. 某类动作对未来回报的延迟影响。

### 9.3.4 注意力池化

编码后的整段序列并非简单取最后一个时刻，而是通过注意力池化提取更有代表性的整体时序特征：

公式（49）：`h_bar(t) = sum( alpha_i * h_i )`，其中 `i` 从 `t-L+1` 到 `t`

其中：

公式（50）：`sum(alpha_i) = 1`

### 9.3.5 Dueling Q 头

在得到序列特征 \(\bar h_t\) 后，网络分为价值流与优势流：

公式（51）：`V(t) = f_v( h_bar(t) )`

公式（52）：`A(t, a) = f_a( h_bar(t), a )`

最终 Q 值计算为：

公式（53）：`Q(t, a) = V(t) + [ A(t, a) - average_over_actions( A(t, a') ) ]`

该设计有助于在动作差异不明显时稳定估计状态价值，在动作差异显著时突出优势动作。

---

## 9.4 Double DQN 目标更新

T-DDQN 的训练采用 Double DQN 更新方式。对于经验样本 \((s_t,a_t,r_t,s_{t+1})\)，目标值定义为：

公式（54）：`y(t) = r(t) + gamma * Q_target( s(t+1), argmax_a Q_eval( s(t+1), a ) )`

其中：

- \(\theta\) 为在线网络参数；
- \(\theta^-\) 为目标网络参数；
- \(\gamma\) 为折扣因子。

损失函数写为：

公式（55）：`L(theta) = E[ ( y(t) - Q_eval( s(t), a(t) ) )^2 ]`

通过最小化式（55），网络逐步学习动作价值函数。

---

## 9.5 T-DDQN 的训练过程

训练流程可描述如下：

1. 初始化在线 Q 网络与目标 Q 网络。
2. 初始化序列经验缓冲区。
3. 环境重置，得到初始状态序列。
4. 在每个时间步：
   1. 根据当前状态序列选择动作；
   2. 环境执行动作，返回奖励与下一状态；
   3. 将样本存入经验池；
   4. 抽样序列批次进行学习；
   5. 周期性或软更新目标网络。
5. 重复上述过程直到达到训练轮数上限。

当前代码中常见的重要超参数包括：

1. 折扣因子 \(\gamma \approx 0.99\)
2. 软更新系数 \(\tau \approx 0.001\)
3. 批大小 `batch_size = 128`
4. Transformer 头数 `nhead = 8`
5. Transformer 层数 `num_layers = 3`
6. 序列长度 `sequence_length = 24`

## 9.6 当前代码实现中的实际特点

这里需要非常严谨地指出一点：  
当前仓库中的 `TDuelingDDQN` 在动作选择函数中加入了较强的启发式动作选择机制。也就是说，从当前执行路径看，它并不是一个完全由神经网络端到端独立决定动作的纯 T-DDQN，而更接近：

**带启发式先验的 T-DDQN 混合决策框架**

其工程含义是：

1. 神经网络仍然承担价值学习和序列表征作用。
2. 但在线动作决策过程中，规则先验起到了显著引导作用。
3. 这种设计通常会提高策略稳定性和工程可解释性。
4. 若用于学术表述，建议明确说明“策略中融合了启发式动作选择机制”，避免把它表述成完全无先验的标准纯深度强化学习模型。

这一点对论文写作尤其重要，因为它关系到方法命名与实现一致性。

## 10. 状态转移与物理执行机制

## 10.1 状态转移本质

项目中的状态转移不是简单的数学迭代，而是由真实含义明确的物理量更新共同决定。设环境状态转移函数为：

公式（56）：`s(t+1) = f( s(t), a(t), xi(t) )`

其中 \(\xi_t\) 包括当前光伏、基础负荷、电价时段等外部输入。

具体而言，执行动作后会导致以下变量更新：

1. 储能 SOC 更新；
2. EV SOC 更新；
3. 空调功率与室温更新；
4. 可平移负荷功率更新；
5. 总负荷与净购电功率更新；
6. 光伏利用率更新；
7. 用户满意度更新。

因此，状态转移本质上是“动作-物理过程-系统反馈”的联动结果。

## 10.2 总负荷合成

在每个时刻，系统总负荷由基础负荷和各可调资源共同叠加得到：

公式（57）：`P_total(t) = P_base(t) + P_ac(t) + P_ev(t) + P_shift(t) + P_bat(t)`

此后，结合光伏消纳情况得到净购电功率：

公式（58）：`P_grid(t) = max( P_total(t) - P_pv_use(t), 0 )`

实际奖励中的经济项、平滑性项和光伏项均由这些量进一步导出。

## 11. 算法流程伪代码

下面给出适合论文或技术报告中的 T-DDQN 调控流程伪代码。

```text
Algorithm 1  基于 T-DDQN 的多柔性负荷协同调控

Input:
    环境 Env
    在线网络 Q_theta
    目标网络 Q_theta_target
    经验回放池 D
    序列长度 L
    折扣因子 gamma
    训练轮数 Episodes

Initialize:
    随机初始化 Q_theta
    令 Q_theta_target <- Q_theta
    初始化经验池 D

for episode = 1 to Episodes do
    重置环境，得到初始状态 s_1
    构造初始状态序列 S_1

    for t = 1 to T do
        根据 S_t 选择动作 a_t
        执行动作 a_t，环境返回 r_t, s_(t+1), done
        更新序列 S_(t+1)
        将 (S_t, a_t, r_t, S_(t+1), done) 存入 D

        if D 中样本足够 then
            采样一个批次的序列样本
            对每个样本计算
                y = r + gamma * Q_theta_target(S', argmax_a Q_theta(S', a))
            计算损失
                L = mean[(y - Q_theta(S, a))^2]
            使用梯度下降更新 Q_theta
        end if

        周期性或软更新 Q_theta_target

        if done then
            break
        end if
    end for
end for
```

在当前项目实现中，如果需要更严格贴合代码执行路径，还应补充一句：在线动作选择可能受到启发式动作选择器的额外干预。

## 附录：2026-05 EV 主逻辑代码同步说明

本附录用于同步 `rural_env.py` 中当前真实生效的 EV 调控逻辑。若正文前文仍保留旧版本“分高光伏 / 中光伏 / 低光伏场景，直接给 `target_power` 赋值”的描述，则应以本附录说明为准。

### A. 当前 EV 目标功率的正式结构

当前代码已将 EV 主逻辑整理为如下结构：

```text
target_power_raw = P_task + Delta_P_pv + Delta_P_prog + Delta_P_peak
```

其中：

- `P_task`：任务底线项；
- `Delta_P_pv`：光伏吸纳修正项；
- `Delta_P_prog`：进度补偿项；
- `Delta_P_peak`：削峰放电修正项。

但实际执行功率并不是直接使用 `target_power_raw`，而是还需经过：

1. 任务存在性与 EV 可用性门控；
2. 功率上下限裁剪；
3. 各算法对应的平滑与单步变化约束。

因此更准确的代码表达应理解为：

```text
target_power = constrained_and_smoothed(target_power_raw)
```

### B. 当前四项与代码的对应关系

#### B.1 任务项 `P_task`

当前代码中：

```text
required_power = remaining_energy / (remaining_steps * 0.25)
P_task = min(max_positive_power, required_power)
```

该项表示：为了在剩余时间内完成当天剩余 EV 任务，当前所需维持的最低任务充电功率。

#### B.2 光伏修正项 `Delta_P_pv`

当前代码中先定义：

```text
pv_ratio = pv_output / day_pv_peak
pv_support = clip((pv_ratio - pv_mid_ratio) / (pv_high_ratio - pv_mid_ratio), 0, 1)
Delta_P_pv = 0.60 * algo_pv_bias * max_positive_power * pv_support
```

其中当前阈值口径为：

- `pv_high_ratio = 0.50`
- `pv_mid_ratio = 0.15`

这表示：当前光伏相对当天峰值越高，EV 越倾向于在任务底线之外增加充电功率，以提升光伏消纳。

#### B.3 进度补偿项 `Delta_P_prog`

当前代码中：

```text
slack = expected_completion - completion_ratio
prog_support = clip(slack / 0.20, 0, 1)
```

若已临近日末且仍有剩余任务，则：

```text
prog_support = max(prog_support, 0.40)
```

最终：

```text
Delta_P_prog = 0.50 * max_positive_power * prog_support
```

该项表示：若 EV 当前任务进度落后于期望完成曲线，则应额外提高当前充电目标；若接近日末，则进一步增强保底补电意愿。

#### B.4 削峰修正项 `Delta_P_peak`

当前代码中，若满足：

```text
current_net_load > (1 + eta_L) * day_mean_net_load
and SOC > 0.50
```

则先计算：

```text
discharge_limit = min(abs(P_ev_min), (SOC - 0.50) * E_ev / 0.25)
net_load_ratio = clip((current_net_load - day_mean_net_load) / (day_peak_net_load - day_mean_net_load), 0, 1)
peak_reduction = min(discharge_limit, k_discharge * abs(P_ev_min) * net_load_ratio)
Delta_P_peak = -peak_reduction
```

其中当前参数口径为：

- `eta_L = 0.10`
- `k_discharge = 0.50`
- 放电 SOC 阈值：`0.50`

该项本质上是一个负向修正项，用于在净负荷显著偏高时把 EV 目标功率向放电方向拉动。只有当其绝对值足够大时，最终 EV 才会由充电状态转入放电削峰状态。

### C. 当前 EV 放电逻辑

当前代码已不再使用旧版本固定绝对值条件：

```text
pv_output < 1.0
current_load > 0.8
SOC > 0.4
```

而是采用：

1. 基于当天净负荷序列构造：
   - `day_mean_net_load`
   - `day_peak_net_load`
2. 计算当前净负荷：
   - `current_net_load`
3. 当 `current_net_load` 显著高于当天基准净负荷，且 `SOC > 0.50` 时，才通过 `Delta_P_peak` 触发放电削峰。

因此当前 EV 放电机制可以归纳为：

```text
基于净负荷超额程度的连续放电修正机制
```

### D. 当前 fallback 机制

当前代码中，EV fallback 已设置为显式开关，且默认关闭：

```text
enable_ev_fallback = False
```

默认关闭的意义在于：

1. 更适合论文实验和算法公平对比；
2. 避免环境隐式补电削弱主策略动作 `action == 3` 的辨识度。

只有显式启用：

```text
enable_ev_fallback = True
```

时，`DDQN` 和 `PPO` 才会在 `action != 3` 且任务未完成时进入保底自动补电逻辑。该机制仅承担保底充电作用，不承担放电削峰作用。

### E. 当前奖励层口径同步

当前 EV 奖励中的光伏判断，也已同步到相对比例口径，即按：

```text
reward_pv_ratio = pv_output / day_pv_peak
```

判断高光伏奖励与低光伏惩罚，而不再沿用固定原始阈值 `5.0 / 1.0 / 1.5`。

因此，当前代码中 EV 的几个关键层面已经形成统一口径：

1. 充电场景：按相对光伏比例判断；
2. 放电场景：按净负荷相对超额程度判断；
3. 奖励层：按相对光伏比例奖励或惩罚；
4. fallback：默认关闭，仅作为可选工程保底机制。
